## 1. 什么是 AI Agent？它和传统 LLM 应用的核心区别是什么？

### 考察点
候选人对 AI Agent 本质的理解，以及是否能清晰区分 Agent 应用与传统 LLM 应用的差异。

### 解答思路

- 从定义入手：AI Agent 是能感知环境、自主规划、调用工具、执行行动并迭代优化的智能系统。
- 对比维度要清晰：从"被动 vs 主动"、"单次 vs 循环"、"封闭 vs 开放"等角度对比。
- 最好能举一个具体例子说明。

### 参考答案

**AI Agent 的定义：**

AI Agent 是以大语言模型为"大脑"，具备感知、规划、工具调用、执行和反思能力的自主系统。它能够接收用户目标后，自主拆解任务、选择工具、执行行动、观察结果、迭代修正，直到达成目标。

**与传统 LLM 应用的核心区别：**

| 维度 | 传统 LLM 应用 | AI Agent |
|------|--------------|----------|
| 交互模式 | 一问一答，被动响应 | 多轮自主循环，主动推进 |
| 执行能力 | 只能生成文本 | 能调用外部工具/API，执行真实操作 |
| 任务处理 | 单次 Prompt 完成 | 自主拆解、规划、分步执行 |
| 决策方式 | 无决策，直接输出 | 有推理-行动-观察的决策循环 |
| 状态管理 | 无状态或简单上下文 | 有记忆系统，跨轮次保持状态 |
| 纠错能力 | 输出即终态 | 能观察结果、反思修正 |
| 确定性 | 相对可控 | 存在非确定性，需要约束机制 |

**关键区别可以用一句话概括：** 传统 LLM 应用是"你说它答"的工具，AI Agent 是"给它目标，它自己想办法完成"的自主体。

**举例：** 传统 LLM 应用像是一个知识渊博的顾问，你问它问题它回答；AI Agent 像是一个实习生，你告诉他"帮我订明天北京到上海最便宜的机票"，它会自己查航班、比价、下单，中间遇到问题还会回来问你确认。

## 2. 你如何定义一个基于 LLM 的智能体？它通常由哪些核心组件构成？

### 考察点
对 Agent 系统架构的理解深度，能否抽象出通用组件模型。

### 解答思路

- 先给出 Agent 的公式化定义
- 按"感知-规划-行动-记忆-反思"的逻辑拆解组件
- 每个组件说明职责、实现方式和关键设计要点

### 参考答案

**定义：**

一个基于 LLM 的智能体 = LLM（推理引擎）+ 记忆（上下文与知识管理）+ 工具（外部能力接口）+ 规划（任务拆解与策略选择）+ 反思（自我评估与优化）

**五大核心组件：**

**1. 大脑（LLM / 推理引擎）**
- 职责：理解意图、推理决策、生成行动序列和回复
- 实现：GPT-4、Claude、Qwen 等通用大模型，或针对特定领域微调的模型
- 关键点：模型的推理能力、指令遵循能力和工具调用能力直接影响 Agent 上限

**2. 记忆系统（Memory）**
- 短期记忆：当前会话的对话历史、中间推理步骤、工具调用结果（通常用上下文窗口或滑动窗口管理）
- 长期记忆：跨会话持久化的用户偏好、历史经验、知识库（通过向量数据库、知识图谱或文件存储实现）
- 工作记忆：当前任务执行过程中的临时状态和中间变量

**3. 工具集（Tools / Skills）**
- 职责：赋予 Agent 执行外部操作的能力
- 形式：API 调用、函数执行、数据库查询、代码运行、文件操作等
- 关键设计：工具描述（让 LLM 理解何时用、怎么用）、参数校验、错误处理、超时与重试

**4. 规划模块（Planning）**
- 职责：将复杂目标拆解为可执行的步骤序列
- 方法：
  - 单步规划：ReAct 式的"思考-行动"循环
  - 多步规划：先生成完整计划再执行（Plan-and-Solve）
  - 动态规划：执行中根据反馈调整计划
- 关键设计：规划粒度控制、子任务依赖管理、失败时的重规划策略

**5. 反思机制（Reflection）**
- 职责：对执行结果进行自我评估，发现问题并迭代改进
- 实现方式：让 LLM 对自身输出进行批判性审查、对比预期与实际结果的差距
- 关键设计：反思触发条件、反思深度控制、防止无限反思循环

**补充理解：**

这五大组件不是孤立存在的，而是通过一个**控制循环（Control Loop）**串联起来：

```
感知（接收输入） -> 规划（生成策略） -> 行动（调用工具） -> 观察（获取结果） -> 反思（评估与调整） -> 循环
```

实际工程中，可以根据场景复杂度做裁剪。简单的客服 Agent 可能只需要 LLM + 工具 + 短期记忆；而复杂的代码 Agent 则需要全部组件。

## 3. 解释 ReAct 框架的工作原理。ReAct 和 Reflexion 有什么区别？

### 考察点
对经典 Agent 范式的理解，能否区分不同框架的设计哲学和适用场景。

### 解答思路

- 先解释 ReAct 的名称来源（Reasoning + Acting）
- 描述 ReAct 的核心循环
- 对比 Reflexion 的"事后反思"机制
- 从流程、时机、开销三个维度做对比

### 参考答案

**ReAct 框架工作原理：**

ReAct = Reasoning（推理）+ Acting（行动），由 Yao et al. (2023) 提出。

核心思想：将推理和行动交织在一起，每一步先"想"再"做"，通过"思考-行动-观察"循环逐步解决问题。

**ReAct 的核心循环：**

```
给定任务 -> Thought（思考当前该做什么）
        -> Action（选择并执行工具）
        -> Observation（观察工具执行结果）
        -> Thought（基于结果继续思考）
        -> Action / Final Answer
        -> ...
```

**一个具体例子（查询天气）：**

```
Task: 北京明天天气怎么样？

Thought: 我需要查询北京的天气预报，使用 weather_api 工具。
Action: weather_api(city="北京", date="明天")
Observation: {"city": "北京", "date": "明天", "weather": "晴", "temp": "15-22°C"}
Thought: 已经获取到天气信息，可以直接回答用户了。
Final Answer: 北京明天天气晴朗，气温 15-22°C。
```

**关键设计要点：**
- 每个 Thought 都是 LLM 对自身推理过程的显式表达（Chain of Thought 的延伸）
- Action 通过结构化的工具描述引导 LLM 选择正确的工具
- Observation 将外部世界的反馈注入到上下文中，影响后续决策

**ReAct vs Reflexion 对比：**

| 维度 | ReAct | Reflexion |
|------|-------|-----------|
| 核心机制 | 推理与行动交替 | 事后反思 + 经验积累 |
| 反思时机 | 每步行动后自然推理 | 任务结束后（或失败时）进行反思 |
| 经验存储 | 不保存经验 | 将反思结果存为长期记忆 |
| 迭代方式 | 单任务内循环 | 跨任务持续优化 |
| 适用场景 | 实时决策、工具调用 | 需要经验积累、多次尝试的任务 |
| 开销 | 每步增加 Thought token | 失败时才触发反思，开销可控 |
| 学习曲线 | 无持续学习能力 | 随使用次数增加而变强 |

**Reflexion 的核心思想：**

Reflexion（Shinn et al., 2023）在 ReAct 的基础上引入了**自我反思**机制。当 Agent 任务失败或结果不理想时，它会：
1. 回顾整个执行轨迹
2. 分析哪里做错了、为什么错了
3. 生成反思总结（"self-reflection"）
4. 将反思存入记忆，下次遇到类似情况时作为启发

这类似于人类的"吃一堑长一智"，让 Agent 具备跨任务的学习能力。

**本质区别一句话：** ReAct 是"边做边想"，Reflexion 是"做完反思、积累经验"。两者不是互斥的，可以组合使用——用 ReAct 做单任务执行，用 Reflexion 做跨任务优化。

## 4. Agent 的记忆系统如何设计？短期记忆和长期记忆怎么区分？

### 考察点
对记忆系统架构设计的实际经验，能否从工程角度给出可落地的方案。

### 解答思路

- 按"三层记忆模型"来组织：工作记忆、短期记忆、长期记忆
- 每层说明存储内容、存储介质、管理策略
- 区分的关键标准：生命周期、容量限制、检索方式

### 参考答案

**三层记忆模型：**

**1. 工作记忆（Working Memory）**

- **内容：** 当前任务执行过程中的中间状态、临时变量、正在处理的数据
- **生命周期：** 任务结束即销毁
- **存储：** 进程内内存、变量、Python 对象
- **容量：** 无硬性限制（受进程内存限制）
- **示例：** ReAct 循环中当前正在遍历的子任务列表、工具调用的中间结果

**2. 短期记忆（Short-term Memory）**

- **内容：** 当前会话的对话历史、上下文信息、近期的工具调用记录
- **生命周期：** 会话级别，会话结束后可以归档或丢弃
- **存储：** 上下文窗口（直接拼入 Prompt）、或会话级缓存（Redis / 内存 DB）
- **容量：** 受限于 LLM 的上下文窗口大小（如 128K、200K tokens）
- **管理策略：**
  - 滑动窗口：保留最近 N 轮对话
  - 摘要压缩：定期对历史对话做摘要，保留关键信息
  - 重要性筛选：基于语义重要性选择性保留
- **示例：** 多轮对话中用户说过的需求、偏好、约束条件

**3. 长期记忆（Long-term Memory）**

- **内容：** 跨会话的用户画像、历史经验教训、领域知识库、工具使用经验
- **生命周期：** 持久化存储，长期有效
- **存储：**
  - 向量数据库（Milvus、Pinecone、Chroma）：语义检索场景
  - 知识图谱（Neo4j）：结构化关系场景
  - 关系数据库：用户画像、配置数据
  - 文件系统：文档、日志
- **容量：** 理论上无上限
- **检索方式：** 语义检索（向量相似度）、关键词检索、图谱查询
- **示例：** 用户的沟通偏好、历史项目中积累的最佳实践、领域 FAQ

**短期 vs 长期记忆的核心区分标准：**

| 标准 | 短期记忆 | 长期记忆 |
|------|----------|----------|
| 生命周期 | 会话级 | 持久化 |
| 容量限制 | 受上下文窗口限制 | 无硬限制 |
| 检索速度 | 即时（在上下文中） | 需要检索（有一定延迟） |
| 更新频率 | 高（每轮对话都更新） | 低（按事件触发更新） |
| 管理复杂度 | 低 | 高（需要索引、去重、过期策略） |
| 典型存储 | Prompt 上下文、Redis | 向量数据库、知识图谱 |

**工程实践建议：**

1. **不要把所有东西都塞进上下文窗口：** 超过 32K tokens 后，模型的注意力会稀释，关键信息可能被忽略
2. **分层检索：** 先快速过滤（短期记忆），再精确检索（长期记忆）
3. **记忆压缩：** 对过期但可能有用的信息进行摘要压缩后再归档
4. **记忆注入策略：** 根据当前任务的相关性，动态决定注入哪些长期记忆到上下文中
5. **防止记忆膨胀：** 设置 TTL、重要性评分、容量上限，避免记忆系统无限增长

## 5. 设计一个支持多工具调用的 ReAct Agent，核心循环是什么？

### 考察点
能否给出一个可实现的 Agent 核心循环伪代码/流程，体现工程化思维。

### 解答思路

- 先画出流程图或伪代码
- 标注每个步骤的输入输出
- 说明关键的工程细节：工具注册、参数解析、错误处理、循环终止

### 参考答案

**核心循环伪代码：**

```python
def react_agent(task: str, tools: List[Tool], max_iterations: int = 10) -> str:
    # 1. 初始化
    messages = [
        SystemMessage(build_system_prompt(tools)),  # 系统提示 + 工具描述
        HumanMessage(task)                            # 用户任务
    ]
    scratchpad = []  # 记录 Thought/Action/Observation 轨迹

    for i in range(max_iterations):
        # 2. LLM 推理：生成 Thought 和 Action
        response = llm.invoke(messages + scratchpad)

        # 3. 解析 LLM 输出
        thought = response.thought
        action = response.action          # 工具名
        action_input = response.params    # 工具参数

        # 4. 判断是否已完成
        if action == "FINISH":
            return response.final_answer

        # 5. 执行工具调用
        tool = find_tool(action, tools)
        if not tool:
            observation = f"错误：未找到工具 '{action}'"
        else:
            try:
                observation = tool.invoke(action_input)
            except Exception as e:
                observation = f"工具执行失败: {str(e)}"

        # 6. 将本轮结果追加到 scratchpad
        scratchpad.extend([
            f"Thought: {thought}",
            f"Action: {action}({action_input})",
            f"Observation: {observation}"
        ])

    # 7. 超时处理
    return "任务未能在规定的迭代次数内完成，请尝试简化任务描述或增加迭代次数。"
```

**核心循环流程图：**

```
[接收任务]
    |
    v
[构建 System Prompt]  ← 包含工具描述、格式约束、行为准则
    |
    v
[LLM 推理]  ← Thought: 分析当前状态，决定下一步
    |
    v
{是 Final Answer?} ──Yes──> [返回结果] ──> [结束]
    |
   No
    |
    v
[解析 Action + Parameters]
    |
    v
[工具路由 & 参数校验]  ← 验证工具是否存在、参数是否合法
    |
    v
[执行工具调用]  ← try/catch 包裹，捕获异常
    |
    v
[记录 Observation]  ← 将工具结果追加到上下文
    |
    v
{达到 max_iterations?} ──Yes──> [超时返回] ──> [结束]
    |
   No
    |
    v
[回到 LLM 推理]  ← 带着新的 Observation 继续循环
```

**关键工程细节：**

1. **工具注册机制：** 每个工具需要提供名称、描述、参数 Schema（JSON Schema），这些信息拼入 System Prompt 让 LLM 理解
2. **结构化输出：** 使用 LLM 的 structured output 能力或 function calling 接口来确保输出格式可控，而不是纯文本解析
3. **错误处理：** 工具调用失败时，将错误信息作为 Observation 返回给 LLM，让 LLM 自行决定重试、换工具还是放弃
4. **循环终止条件：**
   - 正常终止：LLM 输出 Final Answer
   - 异常终止：达到最大迭代次数、检测到循环模式、工具连续失败
5. **上下文管理：** 当 scratchpad 过长时，需要做摘要或截断，避免超出上下文窗口

## 6. 如何避免 Agent 陷入死循环或"踢皮球"？

### 考察点
对 Agent 可靠性问题的实际处理经验，能否给出多层次的防护方案。

### 解答思路

- 死循环和踢皮球是两类问题，分别分析
- 从预防、检测、强制终止三个层次给出方案
- 最好有实际项目中的经验或数据

### 参考答案

**问题定义：**

- **死循环：** Agent 在同一组工具或同一类操作上反复循环，无法推进任务
- **踢皮球：** 在多 Agent 或多工具场景中，不同组件之间来回传递任务，谁都不真正执行

**防护策略（从预防到兜底）：**

**第一层：预防（在设计阶段避免）**

1. **设置最大迭代次数：** 这是最基础的兜底保护，通常设 5-15 次
2. **工具调用次数限制：** 单个工具的调用次数上限，防止"死磕"一个工具
3. **任务复杂度评估：** 在任务开始前评估复杂度，如果超出 Agent 能力范围，直接降级或转人工
4. **工具权限分级：** 高风险工具需要额外确认，防止 Agent 自主执行危险操作

**第二层：检测（在运行阶段识别）**

5. **循环模式检测：**
   - 记录最近 N 步的 (Action, Parameters) 哈希
   - 如果出现重复模式（如相同工具+相同参数出现 3 次以上），判定为死循环
   ```python
   def detect_loop(history: List[Step], window: int = 3, threshold: int = 3) -> bool:
       recent = history[-window:]
       action_signatures = [f"{s.action}:{hash(s.params)}" for s in recent]
       return len(set(action_signatures)) < threshold
   ```

6. **进度评估：**
   - 每轮让 LLM 自我评估："相比上一步，任务是否取得了进展？"
   - 连续 N 轮无进展则触发干预
   - 可以用轻量模型做进度判断，降低成本

7. **语义相似度检测：**
   - 对连续 Thought 做 embedding，如果相似度超过阈值，说明在原地打转

**第三层：干预（检测到后如何处理）**

8. **强制降级：**
   - 死循环时，返回当前已获取的部分结果 + 失败原因
   - 而不是返回空结果或超时错误

9. **切换策略：**
   - 从 ReAct 切换到 Plan-and-Execute（先做完整计划再执行）
   - 或减少工具选择范围，强制使用更直接的路径

10. **人工介入：**
    - 关键业务场景下，死循环时通知人工处理
    - 将 Agent 的执行轨迹一起推送，方便人工理解上下文

**"踢皮球"问题的额外对策：**

- **明确责任边界：** 每个 Agent/工具的职责要清晰，避免模糊地带
- **引入仲裁者（Arbiter）：** 在 Multi-Agent 场景中设置一个协调 Agent，负责判断任务应该由谁执行
- **调用链路追踪：** 记录任务传递路径，检测 A->B->A 的回环模式
- **超时 + 默认行为：** 如果任务被传递超过 N 次还没执行，触发默认行为

**经验数据参考：**

在工程实践中，合理的 max_iterations 设置：
- 简单问答/查询：3-5 次
- 中等复杂度任务（多工具组合）：8-10 次
- 复杂规划任务：15-20 次
- 循环检测窗口：建议 3-5 步

## 7. Plan-and-Execute 和 ReAct 如何选择？

### 考察点
对不同 Agent 范式优缺点的理解，能否根据场景做合理选型。

### 解答思路

- 先分别说明两种模式的原理
- 对比各自的优缺点
- 给出选型决策树或场景对照表

### 参考答案

**两种模式概述：**

**ReAct（边想边做）：**
- 每步只做"思考-行动-观察"，不提前做完整计划
- 优点：灵活、能快速响应环境变化
- 缺点：缺乏全局视角，容易走弯路

**Plan-and-Execute（先计划后执行）：**
- 先由 LLM 生成完整的多步计划，然后逐步执行
- 执行中可以动态调整计划
- 优点：有全局视角，执行路径更优
- 缺点：计划可能不切实际，灵活性差

**选择决策矩阵：**

| 场景特征 | 推荐方案 | 原因 |
|----------|----------|------|
| 任务目标明确、步骤可预期 | Plan-and-Execute | 提前规划最优路径 |
| 环境信息不完整、需要边探索边决策 | ReAct | 无法提前做出可靠计划 |
| 工具数量多、组合方式灵活 | ReAct | 需要动态选择工具 |
| 需要多工具协同、有依赖关系 | Plan-and-Execute | 提前梳理依赖关系 |
| 实时性要求高 | ReAct | 减少计划阶段开销 |
| 任务复杂度高（>5 步） | Plan-and-Execute | 全局规划减少试错 |
| 容错率低（如金融场景） | Plan-and-Execute | 计划可先做人工审核 |
| 环境动态变化 | ReAct 或 混合 | 需要实时适应能力 |

**混合方案（推荐）：**

在实际工程中，最常用的是 **Plan-and-Execute + ReAct 混合模式**：

```
[Plan 阶段] 生成粗粒度计划（高层步骤）
    |
[Execute 阶段] 对每个步骤，用 ReAct 方式执行
    |
    |-- 如果步骤执行顺利 -> 继续下一步
    |-- 如果步骤执行失败 -> 触发 Re-plan（局部重规划）
    |
[完成/失败]
```

这种混合模式的好处：
- 有全局计划作为指引，不走弯路
- 每个步骤保持灵活性，能适应变化
- 失败时只做局部重规划，不需要从头开始

**LangGraph 中的体现：**

```
create_react_agent()     -> 纯 ReAct
create_plan_execute()    -> Plan-and-Execute
自定义 Graph              -> 混合模式（推荐）
```

## 8. Agent 的反思机制如何实现？反思次数怎么控制？

### 考察点
对自我优化机制理解，能否设计出有效的反思策略而非盲目让 LLM 自我检查。

### 解答思路

- 说明反思的触发时机和实现方式
- 反思的 Prompt 设计技巧
- 反思次数的控制策略（硬性限制 + 智能判断）

### 参考答案

**反思机制的实现方式：**

**1. 反思的触发时机：**

- **任务完成后反思（Post-hoc Reflection）：** 无论成功失败都做，用于经验积累
- **任务失败时反思（Failure-triggered）：** 只在失败或结果不满足预期时触发
- **每 N 步反思（Periodic）：** 在长任务中定期暂停做阶段性检查
- **置信度低时反思（Confidence-based）：** LLM 对自身输出置信度低时自动触发

**2. 反思的 Prompt 设计：**

```
你是一个批判性的审查者。请仔细检查以下任务的执行过程和结果：

任务: {task}
执行计划: {plan}
执行轨迹: {trajectory}
最终结果: {result}

请回答以下问题：
1. 任务是否真正完成？结果是否满足所有要求？
2. 执行过程中是否有更优的路径？
3. 有没有遗漏的边界情况或潜在问题？
4. 如果重新执行，你会做出哪些不同的决策？

请以 JSON 格式输出：
{
    "passed": true/false,
    "issues": ["问题1", "问题2"],
    "improvements": ["改进1", "改进2"],
    "retry_needed": true/false,
    "retry_reason": "如果需要重试，说明原因"
}
```

**3. 反思结果的使用：**

- 如果 `passed = false` 且 `retry_needed = true`：触发重执行，将改进建议注入 Prompt
- 无论是否通过，都将反思结果存入长期记忆，作为未来任务的参考
- 反思中提取的模式可以固化为新的工具或工作流

**反思次数的控制策略：**

| 策略 | 说明 | 推荐值 |
|------|------|--------|
| 硬性上限 | 最大反思次数，绝对不能超过 | 2-3 次 |
| 改进递减检查 | 如果连续两次反思的改进建议无实质性变化，停止反思 | 连续 2 次 |
| 置信度阈值 | 反思者给出的评分超过阈值就通过 | 评分 >= 8/10 |
| 成本预算 | 反思消耗 token 超过预算就停止 | 根据场景设定 |
| 边际收益判断 | 如果反思后的改进幅度 < 阈值，停止 | 幅度 < 5% |

**推荐的综合控制逻辑：**

```python
def controlled_reflection(task, result, max_reflections=3):
    for i in range(max_reflections):
        reflection = generate_reflection(task, result)

        if reflection["passed"]:
            break  # 通过，停止反思

        # 检查改进是否有边际效益
        if i > 0 and is_similar_to_previous(reflection, previous_reflections[-1]):
            break  # 改进建议和上次差不多，继续反思意义不大

        # 执行改进并重试
        result = retry_with_improvements(task, reflection["improvements"])
        previous_reflections.append(reflection)

    return result
```

**工程经验：**
- 大多数场景下，2 次反思足够了。第 3 次往往收效甚微
- 反思的 Prompt 要尽量客观，避免 LLM 倾向于"讨好式"地通过
- 可以用一个独立的（较小的）模型来做反思，降低开销
- 反思记录本身就是高质量的数据，可以用于后续微调

## 9. 有微调过 Agent 能力吗？数据集如何收集？

### 考察点
是否有 Agent 微调的实战经验，数据收集方法论是否系统。

### 解答思路

- 说明 Agent 微调的目标（提升工具调用、规划能力等）
- 详细描述数据收集的多种渠道
- 说明数据格式、质量筛选方法
- 给出微调流程和效果评估方式

### 参考答案

**Agent 微调的目标：**

通用大模型在 Agent 场景下往往存在以下不足，需要通过微调增强：
- 工具调用准确率不够高（选错工具、参数格式不对）
- 多步推理能力弱（复杂任务容易迷失方向）
- 输出格式不稳定（JSON 结构不合法）
- 对特定领域知识缺乏理解

**数据集收集方法：**

**1. 自动生成（Synthetic Data Generation）**

- **模板生成：** 定义任务模板 + 工具模板，组合生成大量训练样本
  ```
  任务模板: "帮我查询{城市}{时间}的天气，如果温度高于{阈值}度，就推荐避暑地点"
  工具列表: weather_api, recommend_places
  期望轨迹: Thought -> Action(weather_api) -> Observation -> Thought -> Action(recommend_places) -> ...
  ```
- **模型自生成（Self-Instruct 方法）：** 用强模型（GPT-4/Claude）生成高质量的 Agent 轨迹数据，作为弱模型的训练数据
- **轨迹回放：** 在 Agent 运行过程中记录成功的执行轨迹，自动转换为训练数据

**2. 人工标注**

- 让标注人员模拟用户提出任务，标注正确的工具调用序列
- 对模型生成的轨迹进行人工审核和修正
- 收集真实用户交互中的成功案例

**3. 线上数据挖掘**

- 从 Agent 线上运行日志中提取成功案例
- 对失败案例进行人工修正后加入训练集
- 收集用户的反馈信号（点赞/点踩/修改）

**4. 开源数据集**

- AgentBench、ToolBench 等开源基准
- Gorilla API Bench 等工具调用数据集
- 各平台的 Function Calling 示例数据

**数据格式（以 ReAct 轨迹为例）：**

```json
{
    "task": "查询北京明天天气并推荐出行建议",
    "trajectory": [
        {"role": "thought", "content": "我需要先查询北京的天气"},
        {"role": "action", "content": "weather_api", "parameters": {"city": "北京", "date": "明天"}},
        {"role": "observation", "content": "晴, 15-22°C"},
        {"role": "thought", "content": "天气不错，适合出行，查询推荐景点"},
        {"role": "action", "content": "travel_recommend", "parameters": {"city": "北京", "weather": "晴"}},
        {"role": "observation", "content": "推荐: 故宫、颐和园、长城"},
        {"role": "final_answer", "content": "北京明天晴天，推荐游览故宫、颐和园和长城"}
    ]
}
```

**数据质量筛选：**

1. **去重：** 消除相似的任务和轨迹
2. **多样性控制：** 确保工具组合、任务类型、难度层次的分布均匀
3. **人工抽检：** 对自动生成的数据做抽样审核，标注正确率
4. **困难样本增强：** 对模型容易出错的场景（多工具组合、边界条件）额外生成数据

**微调流程建议：**

- **LoRA 微调**：全量微调成本高，LoRA 在 Agent 能力增强上效果显著
- **训练目标：** 轨迹预测（给定历史步骤，预测下一步）
- **评估指标：** 工具调用准确率、任务完成率、平均步数、轨迹合理性
- **持续迭代：** 线上数据 -> 定期微调 -> 部署验证 -> 循环

## 10. 你用过哪些 Agent 框架？选型是如何选的？最终场景怎么评判好坏？

### 考察点
框架使用的广度和深度，选型方法论是否清晰，效果评估是否系统。

### 解答思路

- 列出主流框架及各自特点
- 给出选型决策维度
- 说明评判标准（效果、效率、可靠性等）

### 参考答案

**主流 Agent 框架对比：**

| 框架 | 特点 | 适合场景 | 学习成本 |
|------|------|----------|----------|
| **LangChain** | 组件丰富、生态成熟 | 快速原型、简单 Agent | 低 |
| **LangGraph** | 图结构控制流、状态机 | 复杂多步 Agent、需要精确控制流 | 中 |
| **AutoGen** | Multi-Agent 原生支持 | 多 Agent 协作、对话式 Agent | 中高 |
| **CrewAI** | 角色驱动、任务编排 | 团队式 Multi-Agent | 低中 |
| **Semantic Kernel** | 微软出品、.NET/Python | 企业级应用、与微软生态集成 | 中 |
| **LlamaIndex** | RAG + Agent 结合 | 知识密集型 Agent | 中 |
| **OpenAI Agents SDK** | 官方出品、轻量 | OpenAI 生态、简单场景 | 低 |
| **Dify / Coze** | 可视化编排 | 非技术人员、快速上线 | 低 |
| **PydanticAI** | 类型安全、简洁 | Python 工程化项目 | 低中 |

**选型决策维度：**

1. **任务复杂度：**
   - 简单单步 Agent -> LangChain / OpenAI Agents SDK
   - 多步有状态 Agent -> LangGraph
   - Multi-Agent 协作 -> AutoGen / CrewAI

2. **团队技术栈：**
   - Python 为主 -> LangChain/LangGraph/AutoGen
   - 需要 .NET/Java 支持 -> Semantic Kernel
   - 前端/全栈 -> Coze / Dify

3. **控制需求：**
   - 需要精确控制执行流程 -> LangGraph（图结构）
   - 灵活即可 -> LangChain（链式）

4. **运维要求：**
   - 需要可视化监控 -> LangSmith + LangGraph
   - 自研监控 -> 轻量框架 + 自定义埋点

5. **部署环境：**
   - 云端 SaaS -> Coze / Dify
   - 私有化部署 -> LangChain / AutoGen / 自研

**评判 Agent 好坏的指标体系：**

| 维度 | 指标 | 说明 |
|------|------|------|
| **效果** | 任务完成率 | 成功完成的任务占比 |
| | 结果准确率 | 结果正确的比例 |
| | 工具调用准确率 | 选择正确工具+正确参数的比例 |
| **效率** | 平均执行步数 | 完成任务平均需要多少步 |
| | 端到端延迟 | 从接收到输出的总时间 |
| | Token 消耗 | 每次任务的平均 token 用量 |
| **可靠性** | 死循环率 | 触发循环保护的频率 |
| | 异常率 | 工具调用失败的比例 |
| | 降级率 | 需要转人工的比例 |
| **体验** | 用户满意度 | 用户评分或反馈 |
| | 可解释性 | 执行轨迹是否易于理解 |

**工程实践建议：**

- 不要一开始就上复杂框架，先用简单方案验证可行性
- 框架选型要考虑长期维护成本，而不是短期开发速度
- 对核心业务逻辑，建议在框架之上做一层封装，降低框架绑定的风险

## 11. 构建复杂 Agent 时最主要的挑战是什么？

### 考察点
是否有实际构建复杂 Agent 的经验，能否识别关键挑战并给出应对策略。

### 解答思路

- 从多个维度列举挑战
- 每个挑战给出具体表现和解决思路
- 重点突出最关键的 2-3 个

### 参考答案

**构建复杂 Agent 的主要挑战（按严重程度排序）：**

**1. 可靠性与确定性控制（最核心挑战）**

LLM 本质上是概率模型，而 Agent 需要在确定性的业务环境中运行。这两个需求之间存在根本矛盾。

- **表现：** 同样的输入，不同次运行结果不一致；工具调用偶尔选错；复杂任务成功率不稳定
- **应对：**
  - 在关键路径上做确定性约束（格式校验、权限控制、白名单）
  - 多层保护：输入校验 -> 过程监控 -> 结果审查
  - 引入确定性 fallback（LLM 失败时走规则引擎）
  - 降低温度参数（temperature=0 或极低）

**2. 可观测性与调试困难**

Agent 的执行链路长、分支多、状态复杂，出问题后很难定位根因。

- **表现：** 用户反馈结果不对，但排查需要看几十步的执行轨迹
- **应对：**
  - 全链路日志记录（每一步的输入、输出、耗时、token 消耗）
  - 可视化的执行轨迹展示（类似 LangSmith 的 trace 视图）
  - 关键节点埋点和告警
  - 录制-回放机制（重现问题场景）

**3. 上下文窗口管理**

随着执行步数增加，上下文越来越长，关键信息被稀释，模型表现下降。

- **表现：** 长任务后期，Agent 忘记早期的重要信息或约束条件
- **应对：**
  - 分层记忆：关键信息持久化到长期记忆，不依赖上下文窗口
  - 动态摘要：定期压缩历史对话，保留核心信息
  - 信息注入策略：按需从记忆中检索相关信息注入上下文
  - 任务分解：将大任务拆成多个小任务，每个任务上下文独立

**4. 工具生态的建设与维护**

工具的数量、质量、描述准确性直接影响 Agent 能力。

- **表现：** 工具描述不清导致选错工具；工具接口变动导致 Agent 失败
- **应对：**
  - 工具描述标准化（名称、功能描述、参数 Schema、使用示例）
  - 工具版本管理和兼容性保证
  - 工具调用的自动化测试

**5. 安全与权限控制**

Agent 具备执行外部操作的能力，安全风险远大于传统 LLM 应用。

- **表现：** Agent 可能执行危险操作、泄露敏感数据、被 Prompt Injection 攻击
- **应对：**
  - 工具权限分级（只读、写入、管理）
  - 敏感操作需要人工确认
  - Prompt 安全过滤
  - 输出审查和内容安全检测

**6. 成本优化**

复杂 Agent 的 token 消耗可能非常高，尤其是多轮循环和反思场景。

- **应对：**
  - 小模型做大模型的路由
  - 缓存重复查询的结果
  - 优化 Prompt 长度
  - 设置成本上限和告警

## 12. Agent 和 RAG 什么关系？什么时候用哪个？

### 考察点
对两个概念的本质理解，能否清晰界定各自的边界和组合方式。

### 解答思路

- 先明确 RAG 和 Agent 各自的定位
- 说明它们是互补而非竞争的关系
- 给出决策树和实际场景分析

### 参考答案

**本质关系：互补而非竞争**

- **RAG（Retrieval-Augmented Generation）：** 解决的是"知识从哪里来"的问题。它让 LLM 能访问外部知识库，回答训练数据之外的知识。
- **Agent：** 解决的是"事情由谁来做"的问题。它让 LLM 能主动规划、调用工具、执行操作，完成复杂任务。

**两者可以组合使用：Agentic RAG = Agent（规划和执行）+ RAG（知识检索）**

**选择决策树：**

```
用户需求是什么类型？
|
+-- 需要基于特定知识回答问题
|   +-- 只回答，不需要执行操作 -> RAG
|   +-- 回答后还需要执行操作 -> Agent + RAG
|
+-- 需要执行外部操作（查API、写文件、发邮件）
|   +-- 不需要查知识库 -> 纯 Agent
|   +-- 需要结合知识库 -> Agent + RAG
|
+-- 需要多步骤、多工具协作完成复杂任务
    +-- 不需要外部知识 -> 纯 Agent
    +-- 需要外部知识 -> Agent + RAG
```

**场景对照：**

| 场景 | 方案 | 原因 |
|------|------|------|
| 企业内部知识库问答 | RAG | 核心需求是检索和回答 |
| 智能客服（FAQ + 订单操作） | Agent + RAG | 既要查知识又要执行操作 |
| 数据分析 Agent（查数据库+生成报告） | Agent | 核心是工具调用和分析 |
| 代码助手（基于代码库回答） | RAG | 检索代码库 + 生成回答 |
| 自动化运维（监控+排查+修复） | Agent | 多工具协作、决策执行 |
| 研究报告生成 | Agent + RAG | 检索文献 + 规划写作 + 生成 |

**Agentic RAG 的典型架构：**

```
用户提问
    |
    v
[Agent 规划模块]  ← 决定是否需要检索、检索什么、检索后做什么
    |
    +-- 需要知识? -> [RAG 检索] -> 注入上下文
    +-- 需要操作? -> [工具调用] -> 执行
    |
    v
[LLM 综合生成最终结果]
```

**一句话总结：** RAG 解决的是信息获取问题，Agent 解决的是任务执行问题。两者不是替代关系，而是可以组合的积木。

## 13. LangGraph 和 LangChain 的本质区别是什么？什么时候用哪个？

### 考察点
对两个框架设计哲学的理解，能否从架构层面区分它们。

### 解答思路

- LangChain 是"链式"，LangGraph 是"图式"
- 从控制流、状态管理、适用场景三个维度对比
- 给出迁移和组合建议

### 参考答案

**本质区别：控制流模型不同**

- **LangChain：** 基于 **Chain（链式）** 模型。任务按线性顺序执行，A -> B -> C，适合流程固定的场景。
- **LangGraph：** 基于 **Graph（图/状态机）** 模型。任务执行流可以用有向图表示，支持循环、分支、条件跳转，适合需要复杂控制流的 Agent。

**详细对比：**

| 维度 | LangChain | LangGraph |
|------|-----------|-----------|
| 控制流 | 线性链式（DAG） | 图结构（支持循环） |
| 状态管理 | 隐式（在链中传递） | 显式（State 对象） |
| 循环支持 | 不支持（需手动实现） | 原生支持 |
| 条件分支 | 有限（通过 Router） | 原生支持（条件边） |
| 人类介入 | 需要额外实现 | 原生支持（interrupt_before/after） |
| 调试能力 | 基础（通过 LangSmith） | 更强（节点级别追踪） |
| 学习曲线 | 低 | 中等 |
| 适合场景 | 简单流水线、RAG | 复杂 Agent、多步决策 |

**代码对比示例：**

LangChain 的链式（线性）：
```python
chain = prompt | llm | output_parser
result = chain.invoke({"question": "..."})
```

LangGraph 的图式（循环）：
```python
def agent_node(state):
    response = llm.invoke(state["messages"])
    return {"messages": [response]}

def should_continue(state):
    if "FINISH" in state["messages"][-1].content:
        return END
    return "tools"

workflow = StateGraph(State)
workflow.add_node("agent", agent_node)
workflow.add_node("tools", tool_node)
workflow.add_conditional_edges("agent", should_continue)
workflow.add_edge("tools", "agent")  # 循环边
app = workflow.compile()
```

**选择建议：**

- **用 LangChain 的场景：**
  - RAG 流水线（检索 -> 重排 -> 生成）
  - 简单的 Prompt 链
  - 不需要循环和复杂状态管理的任务
  - 快速原型验证

- **用 LangGraph 的场景：**
  - ReAct Agent（需要循环）
  - Multi-Agent 协作（需要复杂的路由和状态传递）
  - 需要人类介入审批的工作流
  - 需要精确控制执行顺序和重试逻辑的场景

**实际工程建议：**

两者不是互斥的。**LangGraph 建立在 LangChain 之上**，可以使用 LangChain 的组件（Prompt、Model、Tool、Retriever），但用 LangGraph 来控制执行流。推荐的方式是：

- 底层组件用 LangChain（工具、模型封装、Prompt 管理）
- 控制流用 LangGraph（状态机、循环、条件分支）

## 14. 在 Agent 中，记忆模块你一般会怎么设计？（小红书面经）

### 考察点

与第 4 题类似，但更侧重实战经验和具体实现细节。小红书面经通常期望听到具体的技术选型和踩坑经验。

### 解答思路

- 给出一个完整的技术架构方案
- 包含具体的技术选型
- 分享实践中遇到的坑和解决方案

### 参考答案

**记忆架构设计（生产级）：**

```
                    ┌─────────────────────────────────────┐
                    │         记忆管理层 (Memory Manager)  │
                    │   - 检索策略  - 注入策略  - TTL 管理  │
                    └──────────────┬──────────────────────┘
                                   │
          ┌────────────────────────┼────────────────────────┐
          │                        │                        │
    ┌─────▼──────┐          ┌──────▼──────┐          ┌──────▼──────┐
    │  工作记忆   │          │  短期记忆    │          │  长期记忆    │
    │ (进程内)    │          │ (会话级)     │          │ (持久化)     │
    │            │          │             │          │             │
    │ Python 对象 │          │ Redis       │          │ 向量数据库   │
    │ 临时变量    │          │ 会话缓存     │          │ 知识图谱     │
    │            │          │             │          │ 关系数据库   │
    └────────────┘          └─────────────┘          └─────────────┘
```

**具体实现：**

**工作记忆：**
- 使用 Python 的 `dataclass` 或 Pydantic 模型定义状态对象
- 在 LangGraph 中通过 State 对象管理
- 不持久化，任务结束即释放

**短期记忆：**
- Redis 存储，以 `session_id` 为 key
- 每个会话独立管理，支持并发
- TTL 设置为 24 小时（可配置），过期自动清理
- 实现滑动窗口：只保留最近 N 轮对话，超出部分做摘要后存档
```python
def get_session_memory(session_id: str, max_turns: int = 10):
    messages = redis.lrange(f"session:{session_id}:messages", 0, max_turns*2-1)
    return parse_messages(messages)
```

**长期记忆：**
- **向量存储（语义记忆）：**
  - 使用 Milvus 或 Chroma
  - 存储内容：历史对话摘要、用户偏好、经验教训
  - 检索方式：语义相似度检索（Top-K）
  - 更新策略：新信息增量写入，定期做去重和压缩

- **知识图谱（关系记忆）：**
  - 使用 Neo4j
  - 存储内容：实体关系、领域知识
  - 适合需要推理关系的场景

- **关系型数据库（结构化记忆）：**
  - 存储用户画像、配置参数、统计数据
  - 使用 PostgreSQL

**记忆注入策略（关键设计）：**

```python
def inject_memory(task: str, session_id: str) -> List[Message]:
    messages = []

    # 1. 加载会话记忆（短期）
    session_mem = get_session_memory(session_id)
    messages.extend(session_mem)

    # 2. 检索相关长期记忆
    relevant_memories = vector_db.search(
        query=task,
        top_k=5,
        threshold=0.7
    )

    # 3. 按相关性排序并注入
    for mem in relevant_memories:
        if mem["importance"] > 0.5:  # 只注入高重要性的
            messages.append(SystemMessage(mem["content"]))

    return messages
```

**实战踩坑经验：**

1. **坑：向量检索返回的信息不相关**
   - 解决：在向量检索后加一层 Cross-Encoder 重排，过滤低相关性结果

2. **坑：记忆太多导致 Prompt 超长**
   - 解决：设置记忆预算上限（如 8K tokens），按相关性排序后截取

3. **坑：旧的记忆一直不被清理**
   - 解决：实现记忆衰减机制，基于最后访问时间计算衰减分数，低于阈值自动归档或删除

4. **坑：多个 Agent 共享记忆时出现竞争**
   - 解决：使用分布式锁或乐观并发控制，确保记忆更新的原子性

5. **坑：记忆更新不一致**
   - 解决：引入记忆版本号，写入时做冲突检测和合并

## 15. Agent 的任务规划能力如何提升？有哪些常见方法？

### 考察点
对 Agent 核心能力（规划）的深入理解，能否从多个维度给出系统性的提升方案。

### 解答思路

- 从 Prompt 工程、架构设计、模型增强、外部辅助四个维度展开
- 每个维度给出具体方法和实际效果
- 区分短期可做的（Prompt）和长期要做的（微调）

### 参考答案

**提升 Agent 任务规划能力的系统性方法：**

### 维度一：Prompt 工程（最快见效）

**1. Chain of Thought (CoT)：**
让 LLM 在回答前先展示推理过程，显著提升复杂任务的规划质量。
```
请先一步步思考：这个任务需要哪些步骤？每个步骤需要什么工具？
然后再生成执行计划。
```

**2. Few-shot 示例：**
在 Prompt 中提供 2-3 个高质量的规划示例，让 LLM 学习规划的模式和粒度。
```
示例 1：
任务: "分析最近一个月的销售数据并生成报告"
计划:
  Step 1: 查询销售数据 (工具: sales_api, 参数: last_30_days)
  Step 2: 数据统计分析 (工具: data_analysis, 参数: 上一步结果)
  Step 3: 生成可视化图表 (工具: chart_generator)
  Step 4: 撰写报告文本 (工具: report_writer)
```

**3. 规划模板（Template-based Planning）：**
为常见任务类型定义规划模板，LLM 只需填充参数而不是从零开始规划。
```python
PLANNING_TEMPLATES = {
    "数据分析": [
        ("获取数据", "data_source", {"query": "..."}),
        ("数据清洗", "data_cleaner", {"source": "$step1.output"}),
        ("分析统计", "data_analyzer", {"data": "$step2.output"}),
        ("生成报告", "report_generator", {"results": "$step3.output"})
    ]
}
```

**4. 子任务分解提示：**
显式引导 LLM 将复杂任务拆解为子任务。
```
将以下任务分解为不超过 5 个子任务，每个子任务：
- 目标明确
- 可以被一个工具完成
- 如果有依赖关系，请标注依赖
```

### 维度二：架构设计（中期投入）

**5. 分层规划（Hierarchical Planning）：**
先做高层规划（粗粒度），执行时再对每个高层步骤做细粒度规划。
```
高层计划: [数据获取] -> [数据分析] -> [报告生成]
执行 [数据获取] 时:
  细粒度计划: [连接数据库] -> [执行查询] -> [验证数据完整性]
```

**6. 动态重规划（Re-planning）：**
执行过程中遇到意外情况时，能够局部调整计划而不是从头开始。
```python
def execute_with_replanning(plan, state):
    for step in plan:
        try:
            state = execute_step(step, state)
        except Exception as e:
            # 基于当前状态重新规划
            new_plan = replan(plan, step, state, e)
            plan = new_plan
            state = execute_step(step, state)
    return state
```

**7. 规划-执行解耦：**
将规划模块和执行模块分开，规划用强模型，执行可以用弱模型，降低成本。
```
[强模型] 生成计划 -> [弱模型] 逐步执行 -> [强模型] 评估结果
```

### 维度三：模型增强（长期投入）

**8. SFT 微调：**
使用高质量的规划轨迹数据微调模型，专门增强规划能力。
- 数据：人工标注的规划示例 + 模型自生成的高质量规划
- 目标：让模型学会合理的任务分解、工具选择、依赖管理

**9. DPO/ORPO 偏好优化：**
对比好的规划和差的规划，训练模型偏好正确的规划方式。

### 维度四：外部辅助

**10. 规划验证器（Planning Validator）：**
在计划执行前，用一个独立的验证器检查计划的合理性：
- 工具是否存在
- 参数类型是否匹配
- 依赖关系是否合理（不能有循环依赖）
- 步骤是否可达

**11. 规划缓存：**
对常见任务的规划结果做缓存，避免每次都重新规划。
```python
plan_cache = {}

def get_cached_plan(task_signature: str) -> Optional[Plan]:
    return plan_cache.get(hash(task_signature))
```

**方法对比总结：**

| 方法 | 见效速度 | 成本 | 效果 | 推荐优先级 |
|------|----------|------|------|-----------|
| CoT + Few-shot | 即时 | 低（仅 token 成本） | 中高 | P0 |
| 规划模板 | 1-2 天 | 低 | 高（特定场景） | P0 |
| 分层规划 | 3-5 天 | 中 | 高 | P1 |
| 动态重规划 | 1-2 周 | 中 | 高 | P1 |
| 规划验证器 | 3-5 天 | 中 | 中高 | P1 |
| 规划缓存 | 1-2 天 | 低 | 中 | P2 |
| SFT 微调 | 2-4 周 | 高 | 高 | P2 |
| DPO 偏好优化 | 4-8 周 | 高 | 中高 | P3 |

**推荐路径：** 先用 CoT + Few-shot + 规划模板快速提升 -> 再加分层规划和重规划 -> 最后考虑微调。
