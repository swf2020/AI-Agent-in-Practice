## 1. 大模型部署时 vLLM 的原理是什么？为什么能加速推理？

**考察点**：理解现代 LLM 推理引擎的核心优化机制和性能调优能力。

**解答思路**：
1. 先概述 vLLM 的定位和核心创新（PagedAttention）
2. 分点说明 PagedAttention 的工作原理及其对显存管理的优化
3. 补充 vLLM 的其他加速特性（连续批处理、内核融合等）

**参考答案**：

vLLM 是目前最主流的大模型推理服务框架，其核心创新是 **PagedAttention** 算法，灵感来自操作系统的虚拟内存分页机制。

传统推理框架（如 HuggingFace Transformers）为每个请求的 KV Cache 预分配连续显存块，导致严重的显存碎片化——一个序列即使只用了 60% 的预分配空间，剩余 40% 也无法被其他请求使用，实际显存利用率只有约 20%-40%。PagedAttention 将 KV Cache 切分为固定大小的"页"（block），每个页在显存中非连续存储，通过页表映射逻辑地址到物理地址。当序列增长时，只需按需分配新页，无需预分配大块连续空间。这使得显存利用率可提升至 90% 以上。

在此基础上，vLLM 引入了 **Continuous Batching**（连续批处理），区别于传统的 static batching（等待整批请求全部完成才出结果），它允许请求在生成完成后立即退出批次，同时新请求可以动态插入空出的位置。这大幅提高了 GPU 吞吐率。

其他加速手段包括：CUDA 内核融合（减少 kernel launch 开销）、支持的多种量化（INT8/INT4/AWQ/GPTQ）、投机解码（speculative decoding）支持、以及多 LoRA 同服务部署等。生产实践中，vLLM 相比原生 HuggingFace 推理可实现 2-4 倍的吞吐量提升，延迟降低约 50%。

- 能提到 vLLM 的 GPU 调度策略（tensor parallelism + pipeline parallelism）
- 能对比 vLLM 与 TensorRT-LLM、TGI 的适用场景差异

## 2. KV Cache 是什么？它如何加速推理？显存占用如何估算？

**考察点**：理解自回归解码阶段的计算瓶颈和优化手段，具备显存容量规划能力。

**解答思路**：
1. 解释 KV Cache 的概念和来源（prefill 阶段计算结果的复用）
2. 说明其对加速推理的作用（避免重复计算）
3. 给出显存占用的估算公式和具体计算示例

**参考答案**：

KV Cache 是自回归解码阶段的核心优化技术。在 LLM 推理过程中，每次生成一个 token 时，需要将之前所有 token 作为上下文重新输入模型。KV Cache 将 prefill 阶段（处理用户输入的 prompt）计算出的 Key 和 Value 矩阵缓存起来，在后续的 decode 阶段（逐个生成 token），只需要对新 token 做 QKV 计算，历史 token 的 K、V 值直接从缓存读取，避免了大量重复计算。

KV Cache 加速推理的本质是用**显存换计算**：decode 阶段从 O(n) 的注意力计算降低到 O(1)（每次只计算新 token 的注意力），将解码过程从计算密集型转变为显存带宽密集型，大幅提升了生成速度。

**显存占用估算公式**：

```
KV Cache 显存 = num_layers × 2 × num_heads × hidden_size_per_head × seq_length × batch_size × dtype_bytes
```

以 Llama-3-8B 为例：32 层、32 个 KV heads、每 head 128 dim、dtype 为 FP16（2 bytes），batch_size=64，seq_length=4096：

```
KV Cache = 32 × 2 × 32 × 128 × 4096 × 64 × 2 bytes
         ≈ 68 GB
```

实际生产环境中，KV Cache 往往占据绝大部分显存（模型权重约 16GB FP16，KV Cache 可达 50GB+）。这也是为什么 vLLM 的 PagedAttention 如此重要——它通过动态页分配减少了 KV Cache 的显存浪费。

- 能提到 KV Cache 量化（FP8/INT8 KV Cache）可以进一步减少显存
- 能讨论 sliding window attention、grouped-query attention 对 KV Cache 的影响
- 能说明 prefix caching 在 multi-turn 场景中的复用价值

## 3. 模型量化部署（INT8/INT4）的原理和 trade-off？

**考察点**：掌握模型压缩技术，能在精度和效率之间做生产级权衡。

**解答思路**：
1. 解释量化的基本原理（浮点到整数的映射及反量化）
2. 对比 INT8 和 INT4 的精度损失、加速效果和适用场景
3. 给出生产环境中的选型建议

**参考答案**：

模型量化的核心思路是将模型权重从高精度浮点数（FP16/BF16）映射到低精度整数（INT8/INT4），在反量化时再映射回浮点数参与计算。公式为：

```
W_int8 = round(W_fp16 / scale) + zero_point
W_fp16_approx = (W_int8 - zero_point) × scale
```

**INT8 量化**（8-bit）：将 FP16 的 65536 个可表示值压缩到 256 个。通常精度损失在 1%-3% 以内，几乎不影响模型能力，显存减半，推理加速约 1.5-2x。适合对精度要求较高的生产场景。PTQ（训练后量化）即可达到较好效果。

**INT4 量化**（4-bit）：进一步压缩到 16 个值，显存降至 FP16 的 1/4。精度损失更明显（3%-8%），需要 GPTQ、AWQ、bitsandbytes (NF4) 等高级量化方法。AWQ（Activation-aware Weight Quantization）通过识别激活敏感的权重保留高精度，精度损失显著低于简单量化。

| 维度 | FP16 | INT8 | INT4 (NF4) |
|------|------|------|------------|
| 显存占用 | 100% | 50% | 25% |
| 推理加速 | 1x | 1.5-2x | 2-3x |
| 精度损失 | 0% | 1%-3% | 3%-8% |
| 量化方式 | - | PTQ | GPTQ/AWQ/NF4 |
| 适用场景 | 基线 | 大多数生产 | 资源受限/边缘 |

**生产级 trade-off**：
- 首选 INT8：精度损失小，推理加速可观，PTQ 成本低（只需少量校准数据）
- INT4 需要 QAT（量化感知训练）或高级 PTQ 方法，适合显存极度紧张的场景
- 注意 INT4 的 kernel 兼容性：需要 vLLM/TensorRT-LLM 等支持 INT4 的推理框架
- 对 Agent 场景（多步推理、工具调用），量化误差会累积放大，需格外谨慎

**加分项**：
- 能说明 AWQ 为什么比 GPTQ 好（保护激活敏感的权重不平滑）
- 能提到混合精度策略（对特定层保留 FP16）
- 能讨论量化对 Agent 工具调用准确率的影响

## 4. 如何设计 LLM API 的高并发服务？

**考察点**：具备生产级 LLM 服务架构设计能力，理解高并发的全链路优化。

**解答思路**：
1. 从架构层面说明组件组成（网关 + 推理引擎 + 缓存 + 队列）
2. 逐层说明优化手段（流式输出、批处理、缓存等）
3. 讨论监控和弹性扩展

**参考答案**：

设计高并发 LLM API 服务需要从架构分层入手，核心组件和策略如下：

**1. 接入层**：使用 API Gateway（Nginx/Kong/APISIX）做负载均衡和路由分发，配置限流（token bucket 或 sliding window）、鉴权、请求路由。对于流式输出，网关需要支持 Server-Sent Events (SSE) 或 WebSocket 的透传。

**2. 服务层**：部署 vLLM 作为推理引擎，配置 Continuous Batching 最大化吞吐。采用多实例部署 + 负载均衡，通过 tensor parallelism 拆分大模型到多卡。对于 Agent 场景，将非流式任务（工具调用、规划）和流式任务（生成回复）分离到不同的服务池。

**3. 缓存层**：实现两级缓存——Prompt Cache（vLLM 内置的 prefix caching）复用相同前缀的 KV Cache；语义缓存对高频相似问题直接返回缓存结果，减少模型调用。

**4. 异步队列**：对非实时请求（批量分析、文档处理），使用消息队列（Kafka/Redis Stream）削峰填谷，worker 异步消费处理。

**5. 关键参数调优**：
- `max_num_seqs`：控制并发序列数，过大会 OOM，过小浪费 GPU
- `max_num_batched_tokens`：控制单批 token 数，影响吞吐和延迟的平衡
- `gpu_memory_utilization`：设置 KV Cache 占比（通常 0.85-0.95）

**6. 监控与弹性**：通过 Prometheus + Grafana 监控 GPU 利用率、请求延迟、队列深度等指标，配合 HPA 做自动扩缩容。

## 5. 多 LoRA 适配器如何同时部署和快速切换？

**考察点**：理解参数高效微调的生产化部署方案，掌握多租户/多场景的模型服务设计。

**解答思路**：
1. 说明多 LoRA 同服务部署的原理（共享基座权重 + 动态加载适配器）
2. 解释切换机制和性能影响
3. 给出生产实践中的配置建议

**参考答案**：

多 LoRA 同时部署的核心思路是**共享基座模型权重，动态加载不同 LoRA 适配器**。以 vLLM 为例，基座模型（如 Qwen2.5-7B）加载到 GPU 显存中，多个 LoRA 适配器（每个通常只有几十 MB）按需加载到同一显存空间。

**实现原理**：推理时，基座模型的权重 W 保持不变，每个请求根据其路由标识，动态选择对应的 LoRA 适配器（BA 矩阵）。计算时执行：
```
output = (W + BA) @ x = Wx + B(Ax)
```
由于 LoRA 的 rank 通常很小（8-64），BA 的矩阵乘法开销很低。vLLM 通过 LoRA 请求路由，将同一适配器的请求 batch 在一起执行，减少切换开销。

**快速切换机制**：
- **预加载策略**：将热门 LoRA 适配器常驻显存，冷 LoRA 从存储热加载
- **LRU 缓存**：显存中维护 LoRA 适配器的 LRU 缓存，超出容量时淘汰最少使用的
- **请求路由**：根据请求的 header 或 body 字段识别目标 LoRA，路由到对应实例

| 维度 | 单模型多实例 | 单实例多 LoRA |
|------|-------------|--------------|
| 显存占用 | 每个实例需完整模型权重 | 共享基座，仅额外 LoRA 显存 |
| 延迟 | 固定 | 冷加载有额外开销（~10-50ms） |
| 扩展性 | 线性增长 | 可支持数十个 LoRA |
| 适用场景 | 不同基座模型 | 同一基座不同任务适配 |

**生产实践**：
- 单个 vLLM 实例建议支持 10-20 个 LoRA 适配器
- LoRA rank 建议 8-32，过大增加显存和计算开销
- 监控 LoRA 切换的命中率和延迟分布
- 对 Agent 场景，不同工具使用不同 LoRA，可显著提升工具调用准确率

**加分项**：
- 能讨论 LoRA 权重合并（merge）与动态切换的性能权衡
- 能提到 vLLM 的 `max_loras` 和 `max_lora_rank` 参数配置
- 能说明 LoRA 热更新（不停服更新适配器）的实现方案

## 6. LLM 服务的限流和降级策略怎么设计？

**考察点**：具备生产级服务高可用设计能力，理解 LLM 服务特有的限流和降级模式。

**解答思路**：
1. 说明限流的层次和算法选择
2. 说明降级的多级策略（从模型降级到功能降级）
3. 给出具体的配置和实施建议

**参考答案**：

LLM 服务的限流和降级需要从多维度设计，形成多层防护：

**限流策略**（按优先级从外到内）：

1. **API 层限流**：使用滑动窗口或令牌桶算法，按用户/租户维度设置 QPS 和并发数上限。例如免费用户 10 QPS、付费用户 100 QPS。
2. **GPU 层限流**：vLLM 通过 `max_num_seqs` 和 `max_num_batched_tokens` 控制并发推理数量，超过上限的请求排队等待。
3. **队列限流**：请求队列设置最大深度，超出直接返回 429 Too Many Requests，避免无限制排队导致超时。

**降级策略**（多级梯度，从轻到重）：

| 级别 | 策略 | 触发条件 | 影响 |
|------|------|---------|------|
| L1 | 降低 max_tokens | 响应延迟升高 | 回复变短，核心功能保留 |
| L2 | 切换到小模型 | GPU 利用率 >90% | 质量略有下降 |
| L3 | 启用缓存命中 | 队列深度 > 阈值 | 相似问题直接返回缓存 |
| L4 | 关闭非必要功能 | 系统负载过高 | 停用工具调用/搜索等 |
| L5 | 静态回复 | 服务不可用 | 返回预设兜底回复 |

**关键设计原则**：
- 降级策略需**可观测、可配置、可回滚**，通过配置中心（如 Apollo/Nacos）动态调整
- 限流返回明确的错误码和重试策略（Retry-After header），客户端需实现指数退避重试
- Agent 场景特别注意：工具调用失败时的降级路径（如搜索失败时回退到纯模型知识回答）
- 记录每次降级的触发原因和时长，用于容量规划

- 能讨论基于预测的弹性限流（根据历史负载趋势提前限流）
- 能提到 Token 预算控制（按用户分配每日/每月 token 配额）
- 能说明 Agent 场景中工具调用的降级链设计（主工具 -> 备选工具 -> 模型自答）

## 7. Docker 容器化部署 Agent 系统时，GPU 资源如何分配？

**考察点**：掌握 GPU 容器化技术栈，理解多容器共享和隔离 GPU 资源的方法。

**解答思路**：
1. 说明 Docker 中 GPU 资源的基本分配方式
2. 对比不同隔离级别的方案及其适用场景
3. 给出生产级 Agent 系统的 GPU 分配架构

**参考答案**：

Docker 容器化部署中 GPU 资源分配有以下几个层次：

**1. 整卡分配（独占模式）**
使用 `docker run --gpus '"device=0"'` 或 `nvidia.com/gpu: 1`（K8s）分配整张 GPU。这是最简单也是生产中最常用的方式，适用于需要完整 GPU 性能的推理服务。缺点是一张小模型可能只用到 30% 的 GPU，其余 70% 被浪费。

**2. MIG（Multi-Instance GPU）分割**
NVIDIA A100/H100/H200 支持 MIG 技术，将单张物理 GPU 切割为最多 7 个独立实例，每个实例有独立的显存和计算核心。适合部署多个小模型实例或开发测试环境。配置示例：
```yaml
resources:
  limits:
    nvidia.com/mig-1g.5gb: 1  # 1个计算实例, 5GB显存
```

**3. MPS（Multi-Process Service）共享**
MPS 允许多个进程共享同一 GPU 的计算资源（但不隔离显存），适合显存需求不大但需要并发计算的场景。需要在宿主机启动 `nvidia-cuda-mps-control -d`。

**4. 时间片共享**
多个容器轮询使用 GPU，适合开发环境，不推荐生产使用。

**Agent 系统的 GPU 分配架构**（生产级）：

```
GPU 0 (A100 80GB): 推理服务 vLLM (独占, ~60GB 模型+KV Cache)
GPU 1 (A100 80GB): 
  - MIG 1: Agent 规划服务 (20GB)
  - MIG 2: 工具调用服务 (20GB)
  - MIG 3: Embedding/Rerank 服务 (20GB)
GPU 2-N: 横向扩展推理服务实例
```

**关键注意事项**：
- 推理容器设置 `--gpus` 或 K8s resource limits，确保 OOM 时容器重启而非影响宿主机
- 监控 GPU 显存使用（`nvidia-smi`）、利用率（DCGM exporter）、温度
- Agent 系统中的非 GPU 组件（工具执行、状态管理等）部署在无 GPU 节点上，降低成本
- K8s 中使用 NVIDIA GPU Operator 简化 GPU 设备插件和驱动管理

- 能讨论 GPU 亲和性调度（同一 Pod 的容器尽量在同一 GPU 节点）
- 能提到 GPU 虚拟化方案如 vGPU (NVIDIA) 或开源的 GPU 共享调度 (Volcano)

## 8. Agent 系统的 CI/CD 流程和普通应用有什么不同？

**考察点**：理解 Agent 系统特有的质量保障难点，具备 AI 应用的工程化能力。

**解答思路**：
1. 对比普通应用和 Agent 系统在 CI/CD 上的核心差异
2. 说明 Agent 特有的评测和质量保障环节
3. 给出完整的 CI/CD Pipeline 设计

**参考答案**：

Agent 系统的 CI/CD 与普通应用有本质区别，核心差异在于**输出的非确定性**和**质量评估的复杂性**：

**核心差异对比**：

| 维度 | 普通应用 | Agent 系统 |
|------|---------|-----------|
| 测试确定性 | 输入固定则输出确定 | 同输入可能不同输出 |
| 质量评估 | 单元测试 pass/fail | 需要 LLM-as-Judge 或人工评分 |
| 回归检测 | 代码 diff + 自动化测试 | prompt 变更 + 模型版本 + 效果回归 |
| 发布风险 | 功能 bug 可快速回滚 | 质量下降可能不明显但影响面广 |
| 依赖管理 | 代码 + 第三方库 | 代码 + prompt + 模型 API + 外部工具 + 知识库 |

**Agent 系统的 CI/CD Pipeline**：

```
代码提交 -> 代码扫描/测试 -> Prompt/配置变更检测 -> 自动化评测 -> 质量门禁 -> 灰度发布 -> 线上监控
```

1. **代码层 CI**：与普通应用相同——单元测试、lint、类型检查、安全扫描。Agent 的工具调用逻辑、状态机定义等仍需传统测试。

2. **Prompt 变更管理**：将 prompt 模板纳入版本控制（类似代码），变更时触发专项测试。使用 DSPy 等框架时，还需测试编译后的 prompt 效果。

3. **自动化评测**（核心差异）：
   - **功能测试**：工具调用正确率、格式合规性（JSON schema 校验）
   - **效果评测**：使用黄金数据集（golden dataset）进行批量回归测试，通过 LLM-as-Judge 或指标（exact match、ROUGE、BERTScore）评估
   - **安全测试**：Prompt injection 攻击测试、越权访问测试
   - **性能测试**：延迟、token 消耗、并发能力

4. **质量门禁**：设置效果指标的最低阈值（如工具调用准确率 > 95%），不达标则阻断发布。

5. **灰度发布**：先切 1%-5% 流量，通过 A/B 测试对比新旧版本的效果指标和线上指标（用户满意度、完成率）。

**加分项**：
- 能讨论 Agent 评测数据集的构建和维护策略
- 能提到 prompt 版本与代码版本的关联管理
- 能说明 LLM-as-Judge 的评估偏差问题和缓解方法
- 能讨论 Agent 的混沌工程（工具不可用、模型超时等异常注入测试）

## 9. 如何做 Agent 服务的自动扩缩容？

**考察点**：掌握云原生环境下 Agent 服务的弹性伸缩策略和实现方案。

**解答思路**：
1. 说明 Agent 服务扩缩容的指标选择（区别于普通应用）
2. 说明 HPA/KEDA 等伸缩工具的使用
3. 讨论扩缩容时的状态管理和冷启动问题

**参考答案**：

Agent 服务的自动扩缩容比传统应用更复杂，因为 GPU 实例成本高、冷启动时间长、且请求的负载差异大（不同 prompt 长度的 token 消耗差异可达 10 倍以上）。

**扩缩容指标选择**：

| 指标 | 适用场景 | 阈值建议 |
|------|---------|---------|
| GPU 利用率 | 推理服务 | >80% 扩容, <30% 缩容 |
| 请求队列深度 | 有排队的场景 | >N 个请求等待时扩容 |
| 平均响应延迟 (P99) | 用户体验敏感 | >SLA 目标时扩容 |
| 并发连接数 | 流式输出场景 | >实例上限时扩容 |
| 自定义 Token 吞吐 | Agent 场景 | tokens/sec < 阈值 |

**实现方案（K8s 生态）**：

1. **HPA（Horizontal Pod Autoscaler）**：基于 CPU/GPU 利用率的自动扩缩容。使用 K8s 的 device-plugin 监控 GPU 指标。

2. **KEDA（Kubernetes Event-driven Autoscaling）**：更适合 Agent 场景，可以基于 Prometheus 指标（队列深度、延迟）或外部事件（消息队列长度）触发伸缩。示例：
```yaml
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: agent-scaler
spec:
  scaleTargetRef:
    name: vllm-deployment
  triggers:
  - type: prometheus
    metadata:
      serverAddress: http://prometheus:9090
      metricName: vllm_queue_depth
      threshold: '10'
```

3. **Scale-to-Zero**：空闲时缩容到 0 实例（KEDA 支持），适合非 24 小时服务，节约成本。冷启动时需要模型预热。

**扩缩容关键问题**：

- **冷启动优化**：新实例启动后需加载模型（数十秒到数分钟），需配合就绪探针和预热脚本。可考虑提前预热（预测性扩容）。
- **优雅下线**：缩容时等待正在处理的请求完成（drain），避免中断 Agent 的多步执行流程。
- **状态管理**：Agent 执行过程中的中间状态需持久化（Redis/DB），确保扩容后新实例能接管进行中的任务。
- **成本控制**：GPU 实例成本高，建议非高峰时段缩容，或采用 spot 实例降低成本（配合检查点和自动恢复）。

**加分项**：
- 能讨论基于预测的伸缩（利用历史流量模式预测扩缩容时机）
- 能提到 GPU 实例的竞价实例（Spot）节省策略和故障恢复
- 能说明 Agent 长任务的跨实例迁移方案

## 10. LLM 推理的首字延迟（TTFT）如何优化？

**考察点**：理解 LLM 推理延迟的构成，掌握端到端的首字延迟优化技术。

**解答思路**：
1. 解释 TTFT 的构成（排队 + 网络 + prefill）
2. 逐层说明优化手段
3. 给出生产环境中的综合优化方案

**参考答案**：

TTFT（Time To First Token，首字延迟）是用户从发送请求到收到第一个 token 的时间，直接影响用户体验。TTFT 由以下几个部分组成：

```
TTFT = 网络传输时间 + 网关处理时间 + 排队时间 + Prefill 时间
```

**优化手段（从外到内）**：

**1. 网络层优化**：
- 客户端和服务端部署在同一 Region，减少网络 RTT
- 使用 gRPC 替代 HTTP（内网场景），减少协议开销
- 连接池复用，避免每次请求建立新连接

**2. 排队优化**（最大优化空间）：
- 合理的实例数量和负载分布，确保队列深度可控
- 请求优先级调度：短请求优先（SJF）、VIP 用户优先
- 预热空闲实例，避免冷启动导致的首请求高延迟

**3. Prefill 优化**（核心计算层）：
- **FlashAttention**：使用 FlashAttention-2/3 加速注意力计算，减少 IO 操作
- **Chunked Prefill**：将长 prompt 分段处理，避免单次 prefill 阻塞 decode 阶段
- **Prefix Caching**：对相同或相似前缀的 prompt 复用 KV Cache（如系统 prompt、multi-turn 对话的上下文），vLLM 的 RadixAttention 支持此功能
- **Speculative Prefill**：对常见 prompt 提前预计算 KV Cache

**4. 模型层优化**：
- 使用较小的模型（如 7B vs 72B）可显著降低 prefill 时间
- 模型量化减少计算量（INT8 prefill 比 FP16 快约 1.5x）
- Tensor Parallelism 将模型切分到多卡并行计算

**5. 架构层优化**：
- Prefill-Decode 分离架构（如 Splitwise）：将 prefill 和 decode 分配到不同 GPU 实例，互不干扰
- 对 Agent 场景，系统 prompt 和工具定义的 KV Cache 可常驻缓存

**生产环境综合方案**（典型 TTFT 优化效果）：

| 优化项 | TTFT 改善 | 实施成本 |
|--------|----------|---------|
| 增加实例数 | 排队时间减少 60%-80% | 中（成本增加） |
| Prefix Caching | 重复 prompt 减少 90% | 低 |
| FlashAttention-2 | prefill 加速 2-3x | 低 |
| Chunked Prefill | 长 prompt 降低 P99 50% | 低 |
| Prefill-Decode 分离 | 综合降低 40% | 高（架构改造） |

**加分项**：
- 能讨论 TTFT 与吞吐量的 trade-off（激进 batching 增加 TTFT）
- 能说明 Speculative Decoding 对 TTFT 的影响
- 能提到 Prompt 长度与 TTFT 的关系（prefill 阶段 O(n^2) 复杂度）

## 11. 模型版本管理和灰度发布怎么设计？

**考察点**：具备 AI 服务的版本治理能力，理解模型更新的风险控制和渐进式发布策略。

**解答思路**：
1. 说明模型版本管理的维度（代码 + prompt + 模型权重 + 配置）
2. 说明灰度发布的多阶段策略
3. 给出回滚和质量保障方案

**参考答案**：

模型版本管理比普通应用更复杂，因为它涉及多个维度的版本控制：

**多维版本管理**：

```
版本标识 = 代码版本 + 模型版本(基座) + 适配器版本(LoRA) + Prompt版本 + 配置版本
```

需要为每个完整的服务快照建立版本标识，推荐使用语义化版本：`v2.3.1-qwen2.5-7b-lora-v12-prompt-v8`。

**版本管理最佳实践**：
- **模型注册表**（Model Registry）：集中管理模型权重文件，记录来源、训练数据、评测指标、兼容性信息。可使用 MLflow、Weights & Biases 或自建。
- **Prompt 版本化**：将 prompt 模板存储在 Git 或数据库中，与代码版本关联，支持 diff 和回滚。
- **配置即代码**：所有运行时配置（temperature、max_tokens、top_p 等）纳入版本控制。
- **模型血缘追踪**：记录每个模型版本的训练数据、超参数、评测结果，确保可追溯。

**灰度发布策略（多阶段渐进式）**：

| 阶段 | 流量比例 | 验证内容 | 时长 |
|------|---------|---------|------|
| 内部验证 | 0%（内部测试） | 功能测试、安全测试 | 1-2 天 |
| Canary | 1%-5% | 延迟、错误率、效果指标 | 半天-1 天 |
| 小流量 | 10%-20% | 用户反馈、业务指标 | 1-2 天 |
| 半量 | 50% | A/B 测试对比 | 2-3 天 |
| 全量 | 100% | 全量监控 | 持续 |

**灰度发布的关键设计**：

1. **流量路由**：通过 header、用户 ID、tenant ID 等维度做流量切分，确保同一用户的请求始终路由到同一版本（sticky session）。

2. **效果对比**：灰度期间同时进行 A/B 测试，对比新旧版本的关键指标（工具调用准确率、用户满意度、任务完成率、token 消耗）。

3. **自动回滚**：设置熔断条件——错误率 > 1%、P99 延迟 > 基线 50%、效果指标下降 > 3% 时自动回滚到上一版本。

4. **蓝绿部署 vs 滚动更新**：LLM 服务推荐蓝绿部署——新旧版本同时运行，快速切换流量。滚动更新不适合，因为模型加载时间长，逐实例替换耗时久。

5. **Agent 特有考虑**：Agent 的多步执行流程在版本切换时可能中断。需要确保会话级别版本一致性——一个 Agent 会话内的所有步骤使用同一模型版本。

**加分项**：
- 能讨论多模型共存的流量调度策略（如按 prompt 复杂度路由到不同模型）
- 能提到模型版本的成本对比（新版本是否更省 token / 延迟更低）
- 能说明模型更新的合规和审计流程（金融/医疗场景）

## 差异总结：部署类题目高频对比

| 对比维度 | 传统 Web 服务 | LLM 推理服务 | Agent 服务 |
|---------|--------------|-------------|-----------|
| 计算瓶颈 | CPU/IO | GPU 显存和带宽 | GPU + 网络（外部工具调用） |

| 限流核心指标 | QPS/并发数 | Token 吞吐/GPU 利用率 | Token + 工具调用次数 |
| 降级策略 | 功能开关/静态页 | 模型切换/降低生成长度 | 工具链降级/纯模型回答 |
| 发布风险 | 代码 bug | 效果退化（难检测） | 效果退化 + 工具调用失败 |
| 测试方式 | 单元测试 | LLM-as-Judge + 黄金数据集 | 端到端场景测试 |
| 冷启动时间 | < 5s | 30s - 5min（模型加载） | 30s - 5min + 工具注册 |
