## Q1. 如何让大模型生产合法的 JSON？（结构化输出保证）

**考察点**：考察候选人对大模型可控输出、结构化生成和工程实践的掌握程度。

**解答思路**：
1. 从 Prompt 层面的基础方法说起（JSON Schema + 指令约束）
2. 进阶到 API 级别的原生支持（response_format、grammar / guided decoding）
3. 补充生产级的兜底策略（重试、后处理校验、降级方案）

**参考答案**：

让大模型稳定输出合法 JSON 需要从 Prompt、API、后处理三个层级同时保障。

第一层是 **Prompt 工程**。在 System Prompt 中明确告知模型输出格式，提供 JSON Schema 或 JSON 示例，并强调"仅输出 JSON，不要包含任何额外文字"。配合 few-shot 示例效果更好。但纯靠 Prompt 约束不可靠——模型仍然可能输出 markdown code fence、前后多余文字或格式微偏差。

第二层是 **API 级别的结构化输出**。主流厂商都提供了原生支持：OpenAI 的 `response_format: { "type": "json_schema", "json_schema": {...} }` 可以在推理时强制模型按 Schema 生成 token；Anthropic Claude 支持 `tools` 的 structured output 模式；开源侧可以通过 **Guided Decoding**（如 Outlines、lm-format-enforcer、XGrammar）在采样阶段利用有限状态机（FSM）约束 token 选择，确保每一步都符合 JSON 语法。这种方法从底层保证输出的合法性，是目前生产环境最可靠的方案。

第三层是 **兜底与校验**。无论前面做了多少约束，生产系统仍然需要：（1）用 `jsonschema` 库对输出做严格校验；（2）校验失败时自动重试（携带错误信息回传给模型）；（3）对截断输出做 JSON 补全（尝试修复缺失的括号或引号）；（4）极端失败时走 fallback 路径。

**加分项**：提到 JSON Schema 约束与 Function Calling 的关系——两者底层技术相同，都是通过 grammar 约束 token 采样空间；提到结构化输出对推理延迟的影响（guided decoding 通常带来 5%-15% 的额外延迟，因为需要在每个 step 做 FSM 状态检查）。

## Q2. 大模型训练为什么需要 warm up 阶段？

**考察点**：考察候选人对学习率调度、训练稳定性和大规模分布式训练底层原理的理解。

**解答思路**：
1. 从梯度不稳定的角度解释 warm up 的必要性
2. 说明 warm up 的具体实现方式和常见策略
3. 结合实践给出 warm up 步数的经验选择

**参考答案**：

Warm up 是训练初期将学习率从一个很小的值（通常接近 0）逐步增加到目标学习率的过程。它解决的核心问题是：**训练初期梯度方向不稳定，直接使用目标学习率会导致训练崩溃**。

具体来说，有三个原因需要 warm up：

1. **参数初始化的随机性**。模型初始化后，各层参数的尺度尚未对齐，前几层的梯度方差远大于后续层。如果一开始就使用大学习率，会导致某些层的参数被过度更新，破坏已经合理的初始化状态。

2. **Adam 优化器的偏差修正**。Adam 的一阶矩和二阶矩估计在初始阶段存在严重的低估问题（bias），导致实际学习率远高于预期。Warm up 可以在偏差修正完成前限制有效学习率，避免参数剧烈震荡。

3. **LayerNorm / RMSNorm 的统计量尚未稳定**。Normalization 层的 running statistics 在训练初期不够准确，如果学习率过大，会导致激活值分布剧烈波动，甚至出现 NaN。

常见的 warm up 策略包括 **线性 warm up**（学习率从 0 线性增长到目标值，最常用）和 **余弦 warm up**。在 LLaMA、GPT 等大规模模型训练中，warm up 步数通常设为总步数的 2%-5%。例如 LLaMA-2 70B 使用了 2000 步的线性 warm up。

**加分项**：提到 AdamW 的 warm up 比 SGD 更重要，因为 Adam 的自适应学习率放大了初始偏差；提到某些研究（如 "Warmup is not required for Adam"）表明在特定条件下可以减少或去掉 warm up，但在工业级大模型训练中 warm up 仍然是标配。

## Q3. MOE 架构和 Dense 架构的区别？各自适用什么场景？

**考察点**：考察候选人对前沿模型架构的理解和实际选型能力。

**解答思路**：
1. 解释两种架构的核心差异（参数激活方式）
2. 从计算效率、训练难度、推理延迟等维度对比
3. 给出实际场景选型建议

**参考答案**：

**Dense 架构**中，输入的每个 token 都会经过模型的所有参数（全激活）；**MOE（Mixture of Experts）架构**中，每个 token 只被路由到一部分 Expert（子网络），其余参数不参与计算，实现**条件计算（Conditional Computation）**。

核心差异总结：

| 维度 | Dense 架构 | MOE 架构 |
|------|-----------|---------|
| 参数激活 | 所有参数参与每个 token 计算 | 仅激活部分 Expert（如 8 个中选 2 个） |
| 推理速度 | 取决于总参数量 | 取决于激活参数量，与总参数量解耦 |
| 显存占用 | 模型权重全部加载，显存需求大 | 同样需要加载全部权重，显存需求不减少 |
| 训练难度 | 成熟稳定 | 路由平衡、Expert 负载不均是难点 |
| 扩展性 | 增加参数直接增加计算成本 | 可以增加 Expert 数量而不增加计算成本 |
| 通信开销 | 标准 All-Reduce | 需要跨设备路由，All-to-All 通信成本高 |

**MOE 的适用场景**：
- 需要大参数容量但受限于推理算力预算的场景（如 Claude、Mixtral、DeepSeek-V3）
- 多领域知识覆盖——不同 Expert 可以隐式学习不同领域的知识
- 对吞吐要求高、对首字延迟要求不极端的场景

**Dense 的适用场景**：
- 推理延迟敏感场景（Dense 的确定性路由无额外开销）
- 算力充足、追求稳定性的场景
- 模型规模不太大（<10B）时，Dense 的效率通常优于 MOE

**加分项**：提到 MOE 训练中的关键挑战——**负载均衡损失（Load Balancing Loss）** 用于防止所有 token 被路由到少数 Expert；**Auxiliary Loss Free Load Balancing** 是最新的改进方向；提到 DeepSeek-V3 的 MLA + Multi-Token Prediction + MOE 的组合架构是 MOE 在工业级大模型中的成功案例。

## Q4. ZeRO 优化器的三个阶段分别优化了什么？

**考察点**：考察候选人对分布式训练显存优化技术的深入理解。

**解答思路**：
1. 先说明 ZeRO 要解决的问题（数据并行中的显存冗余）
2. 依次解释三个阶段的优化对象和原理
3. 补充 ZeRO-3 与 FSDP 的关系

**参考答案**：

ZeRO（Zero Redundancy Optimizer）是 DeepSpeed 提出的数据并行显存优化技术，核心思想是**消除数据并行中各 GPU 之间的显存冗余**。标准数据并行中，每个 GPU 都保存完整的模型参数、梯度和优化器状态，造成大量重复。

| 阶段 | 优化对象 | 优化方式 |
|------|---------|---------|
| **ZeRO-1** | 优化器状态（Optimizer States） | 将 Adam 的 momentum 和 variance 分片存储到不同 GPU 上，通信开销最低 |
| **ZeRO-2** | 优化器状态 + 梯度（Gradients） | 在 ZeRO-1 基础上，将梯度也分片存储，进一步减少显存 |
| **ZeRO-3** | 优化器状态 + 梯度 + 模型参数（Parameters） | 将模型参数本身也分片，每个 GPU 只持有部分参数，需要时通过 All-Gather 获取 |

**ZeRO-1**：优化器状态约占训练显存的 60%（Adam 中每个参数需要额外存储两个与参数等大的状态）。ZeRO-1 将这些状态均匀分片到所有 GPU，每个 GPU 只保存一部分。在反向传播完成后，通过 All-Reduce 同步梯度。显存节省约 N 倍（N 为 GPU 数量）。

**ZeRO-2**：梯度在 ZeRO-1 中仍是全量保存的。ZeRO-2 将梯度也分片，每个 GPU 只保存对应分片的梯度。这进一步减少了约 50% 的显存（因为梯度和优化器状态数量级相同）。

**ZeRO-3**：模型参数也分片存储。前向传播时，当前层需要的参数通过 All-Gather 从各 GPU 收集，使用后立即释放；反向传播同理。这使得可以在显存有限的 GPU 上训练超大模型，但通信开销显著增加。

**加分项**：提到 **ZeRO-Infinity** 支持将溢出数据卸载到 CPU 内存或 NVMe；提到 PyTorch 的 **FSDP（Fully Sharded Data Parallel）** 本质上实现了 ZeRO-3 的思想，是业界更常用的原生方案；提到 ZeRO-3 的通信优化策略——**prefetching**（提前异步获取下一层参数）来掩盖通信延迟。

## Q5. FIM（Fill-in-the-Middle）为什么对代码补全重要？

**考察点**：考察候选人对代码大模型训练技术和 IDE 场景的理解。

**解答思路**：
1. 解释 FIM 的训练方式和数据格式
2. 对比标准自回归预训练与 FIM 的差异
3. 说明在代码补全场景中的实际价值

**参考答案**：

**FIM（Fill-in-the-Middle）** 是一种训练数据排列策略，旨在让模型学会根据上下文中间插入内容，而不仅仅是标准的从左到右续写。

标准自回归预训练中，文本按 `A → B → C` 的顺序训练，模型只能做后缀续写。但代码补全的真实场景是：光标位于文件中间，模型需要**同时利用光标前（prefix）和光标后（suffix）的上下文**来生成中间的代码。FIM 通过在预训练阶段模拟这种场景来解决这个问题。

具体做法是将原始代码切分为 prefix、middle、suffix 三段，训练时以 `<PRE> prefix <SUF> suffix <MID> middle` 的格式排列，让模型学习从 prefix + suffix 预测 middle。Starcoder、CodeLlama、DeepSeek-Coder 等代码模型都采用了 FIM 训练。

**为什么 FIM 对代码补全如此重要**：

1. **匹配真实使用场景**。IDE 补全（如 Copilot、Cursor）中，光标后面的代码（函数签名、后续逻辑）是极强的上下文信号。不做 FIM 训练的模型无法有效利用 suffix 信息。

2. **双向上下文理解**。代码补全需要理解前后文——前面的 import 和函数定义、后面的函数调用和返回类型。FIM 让模型在预训练阶段就学会了这种双向推理。

3. **无需额外训练成本**。FIM 只是在预训练数据排列上做变换，不增加额外的模型参数或训练步骤。以 StarCoder 为例，将 10%-30% 的训练数据做 FIM 变换即可显著提升补全效果。

**加分项**：提到 FIM 的两种数据变换模式——**PSM**（Prefix-Suffix-Middle）和 **SPM**（Suffix-Prefix-Middle），不同模型采用不同格式；提到 FIM 和 **Infilling** 是同一个概念的不同叫法；提到代码补全评测基准 **MultiPL-E** 和 **HumanEval-FIM** 专门评估 FIM 能力。

## Q6. Pass@k 评测指标如何反映实际体验？

**考察点**：考察候选人对代码生成评测体系的理解和批判性思维。

**解答思路**：
1. 解释 Pass@k 的定义和计算方式
2. 分析 Pass@k 与实际用户体验的差距
3. 补充实际生产中如何更全面地评估

**参考答案**：

**Pass@k** 是代码生成领域最常用的评测指标。其含义是：对于每个编程题目，模型生成 k 个不同的代码样本（通过调整 temperature 采样），如果其中至少有一个样本能通过所有测试用例，则该题目记为"通过"。Pass@k = 通过题目数 / 总题目数。

**Pass@k 的计算**：为了避免枚举所有 k 个样本（k 大时不可行），通常采用无偏估计公式：
```
Pass@k = 1 - C(n-c, k) / C(n, k)
```
其中 n 是总采样数，c 是通过测试的样本数。

**Pass@k 与实际体验的关系和局限**：

| 维度 | Pass@k 反映的 | 实际体验的差异 |
|------|-------------|-------------|
| 采样次数 | k=100 意味着从 100 次尝试中选最优 | 用户通常只接受前 1-3 个建议 |
| 正确性定义 | 通过全部测试用例 | 用户需要的是"能跑 + 符合意图 + 代码风格好" |
| 交互模式 | 离线一次性评测 | 实际是多轮对话、迭代修改的过程 |
| 任务复杂度 | 独立函数级题目（HumanEval） | 实际是跨文件、依赖外部库的完整项目 |

**关键局限**：
- **Pass@k 随着 k 增大而单调上升**，但 k=100 的高 Pass@k 不代表 k=1 的体验好。用户实际关心的是 Pass@1。
- **不评估代码质量**。能通过测试但代码冗长、命名混乱、缺少注释的代码仍然计为"通过"。
- **测试用例覆盖度有限**。HumanEval 每道题平均约 7 个测试用例，远不足以覆盖所有边界情况。

**生产级评估建议**：除了 Pass@k，还应关注 **Pass@1**（反映首建议质量）、**BLEU/CodeBLEU**（与参考答案的相似度）、**人工评审通过率**（代码可读性和架构合理性）、以及 **端到端任务完成率**（多步 Agent 任务）。

**加分项**：提到 **SWE-bench** 是比 HumanEval 更接近真实场景的评测基准——在真实 GitHub issue 上评估模型能否自动生成修复 PR；提到 Pass@k 的 **无偏估计公式** 展示了数学功底；提到实际生产中应结合 **用户采纳率**（Acceptance Rate）作为线上核心指标。
