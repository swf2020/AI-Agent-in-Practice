# Module 2 - RAG 设计与优化类

## Q1. RAG 是什么？为什么需要 RAG？RAG 解决了大模型的哪些痛点？

### 考察点

理解 RAG 的本质价值以及它在大模型技术栈中的定位。

### 解答思路

1. 定义 RAG（检索增强生成）及其核心思想
2. 逐一说明它解决的三大痛点：知识过时、幻觉、私有数据不可见
3. 对比说明"不改造模型就能获得外部知识"的架构优势

### 参考答案

RAG（Retrieval-Augmented Generation，检索增强生成）是一种将外部检索与 LLM 生成相结合的架构模式。其核心思想是：在 LLM 生成回答之前，先从外部知识库中检索与用户问题最相关的文档片段，并将这些片段作为上下文一并输入给模型，让模型基于检索到的事实进行回答，而非仅依赖训练阶段记忆的参数化知识。

RAG 主要解决了大模型的三个核心痛点：

**第一，知识时效性问题。** LLM 的知识截止于训练数据的时间点。一旦模型训练完成，它的知识就"冻结"了。RAG 通过外挂知识库，让模型能够访问最新信息，知识库可以随时更新而无需重新训练模型。

**第二，幻觉问题。** 大模型在缺乏相关信息时会"编造"看似合理但实际错误的内容。RAG 将生成过程锚定在检索到的真实文档上，大幅降低了幻觉率。实验表明，在有参考文档的约束下，模型的幻觉率可降低 40%-70%。

**第三，私有/领域数据不可见问题。** 企业内部的知识库、文档、FAQ 等不可能用于预训练。RAG 允许在推理时动态注入私有数据，实现了"模型是通用的，知识是专属的"这一理想架构。

相比微调，RAG 的优势在于：更新知识库的成本几乎为零（只需重新索引），信息可追溯（可以标注引用来源），且数据隐私更容易控制（数据不出库，只在推理时注入上下文）。

**加分项**：能提到 RAG 的学术起源（Lewis et al., 2020 年论文），能区分 Naive RAG、Advanced RAG、Modular RAG 三个演进阶段。

## Q2. RAG 和 SFT 微调的区别是什么？什么时候用哪个？

### 考察点

能否在不同技术路线之间做出正确的架构选型决策。

### 解答思路

1. 从原理层面对比两者的机制差异
2. 从成本、时效性、可控性等维度建立对比矩阵
3. 给出明确的决策树和选型建议

### 参考答案

RAG 和 SFT（Supervised Fine-Tuning）是两种完全不同的大模型能力扩展方式，其本质区别在于知识的存储位置和更新机制。

**RAG** 将知识存储在外部向量数据库中，在推理时通过检索动态注入上下文。知识更新只需修改外部文档，无需碰模型。

**SFT 微调** 则是通过带标注的训练数据调整模型权重，将知识"内化"到模型参数中。知识更新需要重新训练和部署新模型。

| 维度 | RAG | SFT 微调 |
|------|-----|----------|
| 知识存储 | 外部数据库 | 模型参数 |
| 知识更新 | 更新文档即可 | 重新训练 |
| 训练成本 | 无训练成本 | 需要标注数据和算力 |
| 知识时效性 | 实时更新 | 有延迟 |
| 可追溯性 | 可标注引用来源 | 无法追溯 |
| 幻觉控制 | 检索约束，幻觉低 | 仍可能幻觉 |
| 适用场景 | 事实性问答、知识库 | 格式/风格学习、任务适配 |
| 数据隐私 | 数据不出库 | 训练数据可能被逆向 |
| Token 成本 | 每次携带上下文 | 无需额外上下文 |

**选型决策树**：
- 如果需要回答基于企业内部文档的问题 --> RAG
- 如果需要模型学习特定输出格式或风格 --> SFT
- 如果知识频繁更新 --> RAG
- 如果需要降低延迟、减少上下文长度 --> SFT（将知识固化到参数中）
- 如果是开放域问答、需要最新信息 --> RAG
- 如果是专业领域语言理解（如法律术语、医疗术语） --> SFT + RAG 组合

**生产实践中，最常见的是 RAG 为主、SFT 为辅的混合方案**：用 SFT 让模型学会理解特定领域的指令格式和输出规范，用 RAG 提供事实性知识。

**加分项**：能提到 RAG 和微调不是互斥关系，而是互补关系；能举出具体业务场景的选型案例。

## Q3. RAG 的完整链路是怎样的？

### 考察点

对 RAG 系统端到端流程的全局理解和细节把控能力。

### 解答思路

1. 按照"离线索引"和"在线检索"两阶段展开
2. 描述每个子步骤的作用和关键技术点
3. 指出链路中的关键优化节点

### 参考答案

RAG 的完整链路可以分为两大阶段：**离线索引阶段**和**在线检索阶段**。

**离线索引阶段（数据准备）**：
1. **文档采集**：从各种数据源（PDF、Word、网页、数据库、API）获取原始文档
2. **文档解析**：将不同格式统一转为纯文本，处理表格、图片、公式等特殊元素
3. **文本分块（Chunking）**：将长文档切分为合适大小的 chunk，常见的有固定大小分块、语义分块、递归分块等策略
4. **向量化（Embedding）**：将每个 chunk 通过 Embedding 模型转为高维向量
5. **索引存储**：将向量及其元数据存入向量数据库，建立索引（HNSW/IVF 等）

**在线检索阶段（用户请求处理）**：
1. **Query 理解与改写**：对用户问题进行意图识别、拼写纠错、查询扩展等
2. **多路检索**：同时进行向量检索（语义匹配）和 BM25 检索（关键词匹配）
3. **结果融合**：使用 RRF 等策略融合多路检索结果
4. **重排序（Reranking）**：用 Cross-Encoder 对候选结果精排
5. **Top-K 筛选**：选取最终 K 个最相关的 chunk
6. **Prompt 构建**：将检索结果组装为系统提示词
7. **LLM 生成**：将 Prompt 送入大模型，生成回答
8. **后处理**：引用标注、安全过滤、格式调整

```
用户 Query --> Query 改写 --> 多路检索 --> 结果融合 --> Rerank --> Top-K 筛选 --> Prompt 组装 --> LLM 生成 --> 返回回答
```

**关键优化点**：每个环节都可以独立优化。生产系统中，Query 改写和 Rerank 通常对最终效果影响最大；Chunk 策略直接影响检索精度；Embedding 模型的选择是基础。

**加分项**：能画出流程图，能指出每个环节可能出现的问题及对策，能提到 Modular RAG 的灵活编排概念。

## Q4. Embedding 的原理是什么？为什么语义相似的文本向量距离近？

### 考察点

对 Embedding 模型底层原理的理解深度，而非仅停留在调用 API 层面。

### 解答思路

1. 从词向量到句子向量的演进讲起
2. 解释对比学习如何训练出语义空间
3. 说明距离度量的数学含义

### 参考答案

Embedding 的本质是将文本映射到一个高维连续向量空间（通常 384-4096 维），使得语义相似的文本在这个空间中距离较近。

**训练原理**：现代 Embedding 模型（如 BGE、E5、GTE 等）基于 Transformer 编码器架构，通过**对比学习（Contrastive Learning）**训练。具体来说：

1. **正样本对**：同一语义的不同表达（如"如何重置密码"和"密码忘了怎么办"）
2. **负样本对**：语义无关的文本对

训练目标是让正样本对在向量空间中的距离尽可能小，负样本对的距离尽可能大。损失函数常用 InfoNCE Loss：

```
Loss = -log(exp(sim(pos) / T) / Σ exp(sim(neg_i) / T))
```

通过大量语料的训练，模型逐渐学会将语义信息编码到向量的各个维度中。

**为什么语义相似 = 距离近？** 因为训练过程本质上是在学习一个"语义度量空间"。模型通过注意力机制捕捉词语和句子的上下文关系，将这些关系编码为向量。在训练完成后，向量的余弦相似度（cosine similarity）就能近似反映文本的语义相似度。

**生产实践中的关键认识**：
- Embedding 模型的质量直接决定检索上限（Garbage in, garbage out）
- 不同 Embedding 模型的向量空间不同，**不可混用**
- 向量维度越高，表达能力越强，但计算成本和存储也越高
- 领域适配很重要：通用 Embedding 模型在垂直领域的效果可能打 6-7 折

**加分项**：能提到具体的对比学习框架（如 SimCSE、In-batch negative），能解释为什么 cosine 比欧氏距离更常用（与向量模长无关，只关注方向），能区分 Bi-Encoder 和 Cross-Encoder 的适用场景。

## Q5. 混合检索（BM25 + Dense）为什么比纯向量检索好？RRF 融合策略是什么？

### 考察点

对检索技术本质的理解，以及多策略组合的工程思维。

### 解答思路

1. 分析 BM25 和 Dense 各自的优势和盲区
2. 说明两者互补的原因
3. 解释 RRF 算法及其优势

### 参考答案

混合检索之所以比纯向量检索好，是因为 **BM25（关键词匹配）和 Dense（语义匹配）捕捉的是不同维度的相似性，具有很强的互补性**。

**BM25 的优势**：
- 精确匹配：对专有名词、术语、产品型号等精确匹配效果好
- 关键词命中：当用户明确知道要查什么时（如搜索错误码 "ERR_4023"），BM25 比向量检索更精准
- 可解释性强：可以看到哪个词匹配了哪个词

**BM25 的盲区**：
- 无法理解语义：搜"手机"不会返回"iPhone"
- 词汇鸿沟问题：不同表达同一含义的词无法匹配

**Dense 的优势**：
- 语义理解："如何忘记密码"能匹配到"密码重置指南"
- 跨语言/跨表达能力匹配

**Dense 的盲区**：
- 对精确关键词不敏感
- 短文本或专有名词效果差

| 特性 | BM25 | Dense (向量检索) |
|------|------|-----------------|
| 匹配方式 | 关键词精确匹配 | 语义相似度 |
| 擅长场景 | 术语、专有名词、错误码 | 自然语言问题、语义理解 |
| 不擅长 | 语义改写、同义词 | 精确关键词、短文本 |
| 可解释性 | 高 | 低 |
| 索引大小 | 小 | 大 |

**RRF（Reciprocal Rank Fusion）融合策略**：
RRF 是一种简单而有效的多路检索结果融合算法。公式如下：

```
RRF(d) = Σ (1 / (k + rank_d))
```

其中 `d` 是文档，`rank_d` 是文档在某路检索结果中的排名，`k` 是常数（通常取 60）。RRF 的核心思想是：**文档在多个检索结果中排名越靠前，融合分数越高**。

RRF 的优点是无需调参、无需归一化、对单路结果的好坏不敏感，是目前最主流的融合策略。

**加分项**：能提到 BM25 的 IDF 加权机制，能解释 k=60 的经验来源，能讨论 alpha 加权融合与 RRF 的对比。

## Q6. 你已经用混合检索了，为什么还要 Rerank？Cross-Encoder 重排的原理？

### 考察点

理解检索链路中各组件的分工，以及精度与性能的权衡。

### 解答思路

1. 说明检索（召回）和重排的分工差异
2. 解释 Bi-Encoder vs Cross-Encoder 的架构区别
3. 分析为什么 Rerank 是值得的性能投资

### 参考答案

混合检索解决的是**召回率**问题，而 Rerank 解决的是**精确率**问题。两者分工不同，不能互相替代。

**为什么混合检索后还需要 Rerank？**

混合检索虽然比单路检索好，但仍然存在一个根本矛盾：**召回阶段的模型必须速度快，所以精度有限**。具体来说：
- 向量检索使用 Bi-Encoder（Query 和 Document 分别独立编码，然后算余弦相似度），无法让 Query 和 Document 做交叉注意力
- BM25 只看词频，不懂语义

这导致召回的前 50-100 个结果中，真正高度相关的可能只占 30%-50%。Rerank 的作用就是从这个较宽的候选池中，用更精细的模型筛选出最匹配的 Top-K。

**Cross-Encoder 重排原理**：

Cross-Encoder 将 Query 和 Document **拼接在一起**输入 Transformer，让模型在注意力层中对两者进行充分交互：

```
[CLS] Query [SEP] Document [SEP] --> Transformer --> 相关性分数
```

与 Bi-Encoder 的区别：

| 维度 | Bi-Encoder (检索用) | Cross-Encoder (重排用) |
|------|---------------------|----------------------|
| 编码方式 | Query 和 Doc 独立编码 | Query 和 Doc 拼接后联合编码 |
| 注意力 | 各自内部注意力 | Query 和 Doc 交叉注意力 |
| 速度 | 极快（可预计算 Doc 向量） | 慢（每次需重新计算） |
| 精度 | 中等 | 高 |
| 适用阶段 | 召回阶段（从百万级筛选到百级） | 重排阶段（从百级筛选到十级） |

**生产实践中的经典策略**：混合检索召回 50-100 个候选 --> Cross-Encoder Rerank 精排 --> 取 Top-5 给 LLM。这个策略通常能将 NDCG@10 提升 15%-30%。

**加分项**：能提到主流 Rerank 模型（如 Cohere Rerank、BGE-Reranker、Jina Reranker），能讨论 Rerank 的延迟开销和优化方案（如只 rerank 前 20 个）。

## Q7. Chunk 策略怎么设计的？chunk size 设多少？父子索引为什么需要？

### 考察点

对 RAG 数据预处理环节的工程经验，这是影响检索效果的隐性关键因素。

### 解答思路

1. 分析不同 Chunk 策略的适用场景
2. 给出 chunk size 的选值逻辑和实验方法
3. 解释父子索引的动机和工作机制

### 参考答案

Chunk 策略是 RAG 系统中**最被低估但影响最大的环节之一**。切分方式直接决定了检索精度和生成质量。

**Chunk 策略设计**：

| 策略 | 说明 | 适用场景 |
|------|------|---------|
| 固定大小分块 | 按字符/token数切分 | 通用文档，快速上线 |
| 递归分块 | 按段落、句子层级递归切分 | 结构化文档，保持语义完整 |
| 语义分块 | 基于主题/语义变化点切分 | 长文、多主题文档 |
| 文档结构分块 | 按标题、表格、列表等结构切分 | Markdown、HTML、PDF |
| 滑动窗口 | 相邻 chunk 有重叠 | 避免信息被切断 |

**chunk size 怎么选？** 没有万能值，但有经验范围：
- **太小（<200 tokens）**：信息碎片化，LLM 拿到的上下文不完整
- **太大（>2000 tokens）**：噪声多，Embedding 被平均化，检索精度下降
- **推荐区间**：512-1024 tokens，overlap 50-150 tokens
- **选择方法**：在业务数据上做网格搜索（256/512/1024/2048），用 Hit Rate 和 MRR 评估

**父子索引（Parent-Child Indexing）的必要性**：

父子索引解决了"小 chunk 检索准，但信息不够；大 chunk 信息全，但检索不准"的矛盾。

**工作机制**：
1. 将文档切为**小的子 chunk**（如 256 tokens），用于 Embedding 和检索
2. 每个子 chunk 关联到一个**大的父 chunk**（如 2048 tokens）
3. 检索时匹配子 chunk，但返回给 LLM 的是父 chunk

这样做的好处：检索阶段用小 chunk 保证精度，生成阶段用大 chunk 保证完整性。通常能将回答质量提升 10%-20%。

**加分项**：能提到 SentenceWindowRetriever（LlamaIndex 的实现），能讨论不同文档类型（代码、法律文书、技术文档）需要不同的 chunk 策略。

## Q8. 向量数据库怎么选？HNSW 和 IVF 索引原理是什么？

### 考察点

对向量数据库生态的了解程度，以及对底层索引算法的理解。

### 解答思路

1. 梳理向量数据库的选型维度
2. 解释 HNSW 和 IVF 的核心原理
3. 给出不同场景的推荐方案

### 参考答案

**向量数据库选型维度**：

| 维度 | 说明 |
|------|------|
| 数据规模 | 万级、百万级还是十亿级？ |
| 功能需求 | 是否需要混合检索、全文搜索、过滤？ |
| 部署方式 | 云托管 vs 自建 |
| 生态集成 | 是否和现有数据库兼容？ |
| 成本 | 存储成本、查询成本 |

主流向量数据库：
- **Milvus**：开源，大规模场景，功能最全
- **Pinecone**：托管服务，易用性好，但闭源
- **Chroma**：轻量级，适合原型开发
- **Qdrant**：Rust 实现，性能好，支持过滤
- **Weaviate**：支持混合检索和模块化
- **Elasticsearch**：已有 ES 基础设施的团队首选

**HNSW（Hierarchical Navigable Small World）**：

HNSW 是目前最流行的向量索引算法。它的核心思想是构建**多层图结构**：
- 最高层：稀疏的图，节点少，跳得远（类似高速公路）
- 最低层：稠密的图，包含所有节点，精确搜索（类似地面道路）
- 查询时从顶层开始，逐层向下缩小搜索范围

**优点**：查询速度快（O(logN)），召回率高，适合实时插入
**缺点**：内存消耗大（每个向量需要存图结构），构建时间长

**IVF（Inverted File Index）**：

IVF 的核心思想是**先聚类再搜索**：
1. 训练阶段：用 K-Means 将向量空间划分为 N 个聚类（Voronoi 单元）
2. 查询阶段：先找到 Query 所属的聚类，只在该聚类的子空间中搜索

**优点**：内存消耗小，适合超大规模数据集
**缺点**：召回率不如 HNSW，需要合理设置聚类数（nlist 参数）

| 维度 | HNSW | IVF |
|------|------|-----|
| 查询速度 | 极快 | 快 |
| 内存消耗 | 高 | 低 |
| 构建速度 | 慢 | 中等 |
| 召回率 | 高 | 中等 |
| 适合规模 | 百万级以下 | 十亿级以上 |
| 实时插入 | 支持好 | 支持差（需定期重建） |

**加分项**：能提到 HNSW 的关键参数（M, ef_construction, ef_search），能讨论 IVF-PQ（乘积量化）的压缩机制，能结合自身项目给出实际选型理由。

## Q9. Top-K 设多少？设大了设小了各有什么问题？

### 考察点

对 RAG 系统核心超参数的调优经验和系统性思维。

### 解答思路

1. 分析 Top-K 的作用和设置逻辑
2. 分别说明设大和设小的问题
3. 给出动态 Top-K 的方案

### 参考答案

Top-K 是 RAG 系统中**最直接也最关键的超参数之一**，它决定了送入 LLM 的上下文窗口大小。

**Top-K 设小了的问题**：
- **信息不足**：检索到的文档不足以覆盖问题的所有方面
- **关键信息遗漏**：可能把真正相关的文档排在了 K+1 的位置
- **回答不完整**：LLM 基于不完整的上下文生成的回答容易遗漏要点

**Top-K 设大了的问题**：
- **噪声干扰**：不相关的文档混入上下文，LLM 容易被误导
- **上下文窗口占用**：大量无关文本挤占宝贵的上下文空间
- **注意力稀释**：LLM 的注意力机制会在大量文档中分散，降低对关键信息的关注
- **Token 成本**：输入 Token 增多，成本线性增长
- **延迟增加**：LLM 处理更长的上下文需要更多时间

**经验设置**：
- 简单问答（单点事实）：K=3-5
- 复杂问答（需要综合多个来源）：K=5-10
- 对比/总结类问题：K=10-15

**生产实践建议**：
1. 不要设固定 K，采用**动态 Top-K**：根据检索结果的相关度分数自动截断。例如设置一个相似度阈值，只返回分数高于阈值的文档
2. 使用**累积覆盖率**策略：持续添加文档直到覆盖率达到某个百分比
3. 在验证集上评估不同 K 值的效果，找到最优平衡点

```
def dynamic_top_k(results, threshold=0.75, max_k=15):
    return [r for r in results[:max_k] if r.score >= threshold]
```

**加分项**：能提到 NDCG/MRR 等指标来评估不同 K 的效果，能讨论 Rerank 后 Top-K 可以设得更小（因为精度提高了）。

## Q10. RAG 检索到了正确信息，LLM 还是编造了不存在的内容，怎么办？

### 考察点

对 LLM 行为特征的理解和幻觉控制能力。

### 解答思路

1. 分析幻觉的根因
2. 从 Prompt、架构、后处理三个层面给出解决方案
3. 给出具体的实施步骤

### 参考答案

这是 RAG 系统中最常见也最棘手的问题——**检索正确但生成错误**。根因在于 LLM 的参数化知识（训练记忆）与外部上下文之间存在竞争关系。

**解决策略分为三个层次**：

**第一层：Prompt 工程**
- 加入强约束指令："请仅基于以下参考资料回答，不要使用你的内部知识。如果参考资料中没有答案，请明确告知用户。"
- 提供格式示例，展示什么是"好的回答"和"不知道的回答"
- 使用结构化 Prompt，将参考资料和指令清晰分隔

```
你是一个专业的问答助手。请仅根据以下参考资料回答问题。
如果参考资料中没有相关信息，请直接回答："根据现有资料，无法回答此问题。"

【参考资料】
{context}

【用户问题】
{question}
```

**第二层：架构层面**
- **降低 temperature**：将 temperature 设为 0 或接近 0，减少随机性
- **引用强制**：要求 LLM 在回答中标注引用来源，无法标注的内容不予输出
- **两阶段生成**：第一阶段让 LLM 判断参考资料是否足够，第二阶段再生成回答
- **Fact-checking 后处理**：生成后用另一个模型验证每个陈述是否有引用支撑

**第三层：模型层面**
- 选用幻觉率低的模型（如 Claude、GPT-4 比 GPT-3.5 好）
- 对特定领域做 SFT，让模型学会"不知道就说不知道"
- 使用专门优化过 RAG 场景的模型（如 Llama-3 的 instruct 版本）

**调试方法**：用一批标准问题做 A/B 测试，逐条分析幻觉案例，找出幻觉模式（是知识冲突？还是过度推理？还是 Prompt 不够强？），针对性优化。

**加分项**：能提到 Self-Consistency、FactScore 等幻觉检测方法，能讨论 RAG 中"知识冲突"（Parametric vs Contextual Knowledge）的研究。

## Q11. RAG 项目的检索准确率怎么评估？效果不好的时候怎么优化？

### 考察点

对 RAG 系统评估体系的完整认知和问题排查能力。

### 解答思路

1. 区分"检索评估"和"生成评估"两个层面
2. 给出检索阶段的量化指标
3. 给出系统化的优化路径

### 参考答案

RAG 系统的评估需要分层进行，不能只看最终回答质量。

**检索层评估指标**：

| 指标 | 说明 | 计算方式 |
|------|------|---------|
| Hit Rate@K | Top-K 中是否包含相关文档 | 二值判定（有/没有） |
| MRR（Mean Reciprocal Rank） | 第一个相关文档的排名 | 1/rank |
| NDCG@K | 考虑排名和相关度的综合指标 | 归一化折损累积增益 |
| Precision@K | Top-K 中相关文档的比例 | 相关数 / K |
| Recall@K | 所有相关文档中被召回的比例 | 召回相关数 / 总相关数 |

**生成层评估指标**：
- **Faithfulness（忠实度）**：回答是否基于检索到的上下文
- **Answer Relevance（回答相关性）**：回答是否切题
- **Context Recall（上下文召回率）**：参考答案的信息在检索上下文中的覆盖率
- **Context Precision（上下文精确率）**：检索上下文中与答案相关的比例

常用评估框架：Ragas、DeepEval、ARES。

**效果不好的系统化优化路径**：

```
1. 分析问题类型分布 --> 哪类问题效果差？
2. 检查 Chunk 质量 --> 切分是否合理？关键信息被切断了吗？
3. 检查 Embedding 模型 --> 换更强的模型，或做领域微调
4. 检查检索策略 --> 加入 BM25？调整 Top-K？
5. 加入 Rerank --> Cross-Encoder 精排
6. 优化 Query --> 查询改写、扩展
7. 优化 Prompt --> 更强的指令约束
8. 优化模型 --> 换更强的 LLM
```

## Q12. 混合检索权重怎么调？向量检索和 BM25 各占多少？

### 考察点

对检索策略参数调优的实际操作经验。

### 解答思路

1. 说明不同融合策略下的权重概念
2. 给出调参方法和经验值
3. 强调场景依赖性

### 参考答案

混合检索的权重调优是 RAG 系统中需要实验验证的环节，但有一些通用原则。

**RRF 融合（主流方案）**：
RRF 本身不需要显式设置权重，它通过排名倒数自然融合。但可以引入 alpha 参数做加权：
```
Score(d) = alpha * RRF_bm25(d) + (1-alpha) * RRF_dense(d)
```
经验上 **alpha=0.5（等权重）** 是一个好的起点，然后根据业务调整。

**Alpha 加权融合方案**：
如果底层支持设置 alpha（如 Weaviate、Milvus），alpha 的含义是向量检索的权重。

| alpha 值 | 向量权重 | BM25 权重 | 适用场景 |
|----------|---------|----------|---------|
| 0.9-1.0 | 高 | 低 | 语义理解为主的问题 |
| 0.5-0.7 | 中 | 中 | 通用场景（推荐起点） |
| 0.0-0.3 | 低 | 高 | 精确关键词匹配为主 |

**调参方法**：
1. **构建验证集**：准备 50-200 个标注好的 Query-Document 对
2. **网格搜索**：在 [0.0, 0.1, ..., 1.0] 上遍历 alpha
3. **评估指标**：用 NDCG@10 或 MRR 评估每个 alpha 的效果
4. **分类调参**：如果用户问题可以分类（如"搜索术语" vs "自然语言问题"），可以为不同类别设置不同的 alpha

**经验法则**：
- 知识库以技术文档、术语为主 --> alpha 偏小（0.3-0.5）
- 知识库以 FAQ、自然语言问答为主 --> alpha 偏大（0.6-0.8）
- 混合类型（大多数生产场景） --> alpha=0.5

**加分项**：能讨论动态 alpha 选择（根据 Query 特征自动选择权重），能提到基于学习的融合方法（如用一个小模型学习最优权重组合）。

## Q13. Query 改写是怎么做的？有哪些常见方法？

### 考察点

对检索前置优化的理解，这是提升检索效果的高性价比手段。

### 解答思路

1. 解释 Query 改写的作用和必要性
2. 列举主流改写方法
3. 结合实践给出方案选择

### 参考答案

Query 改写的核心目标是**缩小用户原始问题与知识库文档之间的表达差距**，让检索系统能更好地理解用户意图。

**为什么需要 Query 改写？** 用户的问题往往是不完整的、口语化的、或者缺少上下文。直接拿原始 Query 去检索，效果通常不如改写后。

**常见 Query 改写方法**：

| 方法 | 说明 | 示例 |
|------|------|------|
| **拼写纠错** | 修正拼写错误 | "如合重置密码" --> "如何重置密码" |
| **查询扩展（Expansion）** | 添加同义词、相关词 | "电脑蓝屏" --> "电脑蓝屏 死屏 BSOD" |
| **查询分解（Decomposition）** | 将复杂问题拆为子问题 | "A和B的区别是什么？" --> ["A是什么", "B是什么", "比较A和B"] |
| **Step-back Prompting** | 提取问题的高层概念 | "GPT-4 context window 多大" --> "语言模型的上下文窗口" |
| **HyDE（假设文档嵌入）** | 让 LLM 先生成一个假设性答案，再用答案去检索 | 问题 --> LLM生成假设答案 --> 用假设答案的向量检索 |
| **上下文补全** | 利用对话历史补全省略信息 | "它怎么用？"（上文提到某产品） --> "XX产品怎么用？" |
| **意图识别 + 模板化** | 识别意图后套用查询模板 | "对比X和Y" --> 结构化查询 |

**生产实践中的推荐组合**：
1. **第一层（低成本高收益）**：拼写纠错 + 上下文补全
2. **第二层（中等成本）**：查询扩展（用同义词词典或 LLM 生成）
3. **第三层（高成本高收益）**：Step-back Prompting + HyDE

**实现建议**：Query 改写本身可以调用一个小模型（如 GPT-3.5 或 7B 开源模型），成本低、延迟小，但能大幅提升检索效果。

```python
QUERY_REWRITE_PROMPT = """
将以下用户问题改写为更适合检索的形式：
1. 补充缺失的上下文
2. 将口语化表达转为正式用语
3. 保持原意不变
原始问题：{query}
改写后："""
```

**加分项**：能提到 HyDE 的原理和适用场景（对生成式问答特别有效），能讨论多轮对话中的 Query 改写（考虑对话历史）。

## Q14. 多跳检索（Multi-hop RAG）怎么实现？

### 考察点

对复杂检索场景的处理能力，以及推理与检索结合的设计思维。

### 解答思路

1. 解释多跳检索的概念和必要性
2. 给出几种主流实现方案
3. 分析各自的优劣和适用场景

### 参考答案

多跳检索（Multi-hop Retrieval）指的是**需要多次检索才能回答的问题**。用户的问题无法通过一次检索得到完整答案，需要先检索一个信息，基于这个信息再做下一轮检索。

**典型场景**：
- "张三开发的项目和李四开发的项目有什么共同点？" --> 需要先分别查张三和李四的项目
- "A 公司的 CEO 之前在哪家公司工作过？" --> 先查 CEO 是谁，再查该人的履历

**实现方案**：

**方案一：迭代式检索（Iterative Retrieval）**
```
Step 1: 检索问题 P1 --> 得到结果 R1
Step 2: 基于 R1 构造新查询 P2 --> 检索得到 R2
Step 3: 综合 R1 + R2 生成回答
```
优点：逻辑清晰，适合线性依赖的问题
缺点：需要预设跳数，串行执行延迟高

**方案二：自反思检索（Self-Ask / IRCoT）**
让 LLM 判断是否需要进一步检索：
```
LLM: "要回答这个问题，我需要先知道..."
--> 生成子问题 --> 检索 --> 回答子问题 --> 继续推理
--> 当 LLM 判断信息足够时，停止检索
```
优点：自适应，不需要预设跳数
缺点：需要 LLM 有较强的推理能力

**方案三：图谱辅助检索**
将知识构建为图谱，通过图遍历实现多跳：
```
问题 --> 识别实体 --> 在图谱上多跳遍历 --> 收集路径上的信息 --> 生成回答
```
优点：天然支持多跳，可解释性强
缺点：需要构建和维护知识图谱

**方案四：Plan-and-Solve**
让 LLM 先生成检索计划，再并行或串行执行：
```
LLM 生成计划: [查A, 查B, 比较A和B]
并行执行查A和查B --> 收集结果 --> 比较 --> 回答
```

**加分项**：能提到 HotpotQA 等多跳问答数据集，能讨论 IRCoT（Iterative Retrieval Chain-of-Thought）的具体实现，能分析各方案的延迟 trade-off。

## Q15. Self-RAG 和 CRAG 等自适应检索机制是什么？

### 考察点

对 RAG 前沿研究的理解，以及自适应系统设计的思维。

### 解答思路

1. 解释自适应检索的核心思想
2. 分别介绍 Self-RAG 和 CRAG
3. 对比分析与传统 RAG 的区别

### 参考答案

自适应检索的核心思想是：**不是所有问题都需要检索，也不是检索一次就够了。系统应该根据问题的特性和检索结果的质量，动态决定是否需要检索、检索几次、如何使用检索结果。**

**Self-RAG（Self-Reflective RAG）**：

Self-RAG 引入了多个"反射令牌"（Reflection Tokens），让模型在生成过程中自我评估：

1. **Retrieve Token**：模型自主判断是否需要检索
2. **IsRel Token**：评估检索到的文档是否与问题相关
3. **IsSup Token**：评估生成的回答是否被文档支撑
4. **IsUse Token**：评估回答是否有用

训练方式：通过特殊数据让模型学会生成这些反射令牌，形成自我评估循环。

```
Query --> 模型判断: 需要检索? --> 是 --> 检索
--> 模型判断: 文档相关? --> 是 --> 生成回答
--> 模型判断: 回答有支撑? --> 是 --> 输出
```

**CRAG（Corrective RAG）**：

CRAG 的核心是**检索结果质量评估 + 纠正机制**：

1. 检索后，用一个轻量评估器对检索结果打分
2. 如果质量高（分数 > 阈值）--> 直接使用
3. 如果质量中（分数在阈值附近）--> 进行知识修正（结合模型内部知识）
4. 如果质量低（分数 < 阈值）--> 触发 Web 搜索等外部知识源

```
检索结果 --> 评估器 --> 高置信: 直接用
                     --> 中置信: 修正后使用
                     --> 低置信: 搜索外部知识
```

| 机制 | 核心思想 | 何时检索 | 如何处理低质量检索 |
|------|---------|---------|------------------|
| 传统 RAG | 固定流程 | 每次都检索 | 无法处理 |
| Self-RAG | 自我反思 | 模型自主判断 | 通过反射令牌过滤 |
| CRAG | 质量评估 + 纠正 | 每次检索，但结果分三级处理 | 降级到外部搜索 |

**加分项**：能讨论训练 Self-RAG/CRAG 所需的特殊标注数据，能分析这些机制在生产环境中的延迟影响，能提到 FLARE（Forward-Looking Active RAG）等其他自适应方案。

## Q16. 文档更新了，向量索引怎么更新？知识库时效性怎么维护？

### 考察点

对 RAG 系统运维和生命周期管理的理解。

### 解答思路

1. 说明向量索引更新的挑战
2. 给出不同粒度的更新策略
3. 讨论增量更新和全量更新的取舍

### 参考答案

知识库的时效性维护是 RAG 系统从原型走向生产的关键挑战之一。

**向量索引更新的三种策略**：

**策略一：实时更新（Insert/Delete）**
- 新增文档：实时 Embedding 并插入向量库
- 删除文档：按 metadata 或 ID 删除对应向量
- 更新文档：先删除旧向量，再插入新向量

适用场景：文档量不大（百万级以下），对时效性要求高
优点：实时生效
缺点：HNSW 索引在频繁插入时性能会退化，需要定期重建

**策略二：批量重建（定期全量更新）**
- 设定更新周期（如每天/每周）
- 全量重建索引

适用场景：文档更新不频繁，或者对时效性要求不高
优点：索引质量最优
缺点：有更新延迟，重建期间可能影响服务

**策略三：增量 + 定期重建结合**
- 新增文档实时更新
- 修改/删除文档累积到批次后统一处理
- 每周/每月全量重建一次索引以保证质量

**生产实践中的推荐方案**：

```
实时层（热数据）：
  - 最近 7 天的文档，实时插入
  - 使用单独的 collection，查询时合并结果

历史层（冷数据）：
  - 7 天以上的文档，定期重建索引
  - 优化存储，使用 IVF 或 PQ 压缩
```

**文档版本管理**：
- 为每个 chunk 记录版本号、更新时间、来源文档
- 支持回滚到历史版本
- 对过期文档设置 TTL（Time-to-Live）

**关键注意事项**：
- 向量数据库的软删除 vs 硬删除：软删除标记后查询过滤，硬删除立即释放空间
- 索引重建期间的双写策略：新旧索引并行，切换时原子替换

**加分项**：能讨论 CDC（Change Data Capture）与向量索引更新的集成，能提到 Milvus/Pinecone 的具体 TTL 机制。

## Q17. RAG vs Long Context LLM：什么时候该用哪个？

### 考察点

对大模型技术发展趋势的理解和架构选型能力。

### 解答思路

1. 对比两者的技术本质
2. 建立多维度对比矩阵
3. 给出融合使用的建议

### 参考答案

RAG 和 Long Context LLM（长上下文模型，如 200K/1M context window）是两种不同的知识注入方式，各有优劣。

**本质区别**：
- **RAG**：先检索，再注入，知识按需加载
- **Long Context**：一次性加载大量上下文，让模型自己找答案

| 维度 | RAG | Long Context LLM |
|------|-----|-----------------|
| 知识获取 | 检索后注入 | 全量加载 |
| 上下文长度 | 仅加载相关片段 | 全量（可达 1M tokens） |
| 精度 | 取决于检索质量 | 取决于模型的 NIAH 能力 |
| Token 成本 | 低（只传相关片段） | 高（全量加载） |
| 延迟 | 检索延迟 + 生成延迟 | 仅生成延迟（但长上下文生成慢） |
| 精度衰减 | 无衰减 | NIAH（Needle in a Haystack）问题 |
| 可追溯性 | 天然支持 | 较难 |
| 知识更新 | 实时更新 | 需要重新加载 |
| 适用数据量 | 百万级文档 | 通常单次处理量有限 |

**选型建议**：

**选 RAG 的场景**：
- 知识库规模大（十万级以上文档）
- 需要高精度检索（不能丢失关键信息）
- 成本控制敏感（Token 成本重要）
- 需要引用来源和可追溯性

**选 Long Context 的场景**：
- 需要理解的文档本身就不太长（如 50-100 页的报告）
- 需要在上下文中做跨文档的复杂推理
- 检索无法准确找到相关信息（如需要理解全文结构）
- 延迟敏感，不想加检索环节

**融合方案（最佳实践）**：RAG 粗筛 + Long Context 精读。先用 RAG 从海量文档中选出 5-10 篇相关文档，然后一次性加载到 Long Context 模型中进行深度理解。这样既控制了上下文规模，又利用了长上下文的推理能力。

**加分项**：能提到 NIAH（Needle in a Haystack）测试和长上下文模型的精度衰减曲线，能讨论 Lost in the Middle 现象。

## Q18. 多模态 RAG：文档里有图片、表格怎么办？

### 考察点

对多模态信息处理的前瞻思维和实际落地经验。

### 解答思路

1. 分析图片和表格在 RAG 中的处理挑战
2. 分别给出表格和图片的处理方案
3. 讨论多模态 Embedding 的最新进展

### 参考答案

实际的企业文档中，大量关键信息以图片和表格的形式存在，纯文本 RAG 会丢失这些信息。

**表格处理**：

表格是 RAG 中最常见也最难处理的结构化数据。

| 方法 | 说明 | 优缺点 |
|------|------|--------|
| 转为 Markdown | 将表格转为 Markdown 格式，作为文本 chunk | 简单，但复杂表格易丢失结构 |
| 转为 JSON | 将表格转为 JSON 或 CSV 格式 | 保持结构，但 Embedding 效果差 |
| Table QA 模型 | 用专门的表格理解模型（如 TAPAS） | 精度高，但需要额外模型 |
| 表格摘要 | 为每个表格生成文本摘要，对摘要做 Embedding | 平衡方案，推荐 |
| 行列分块 | 按行或按列切分，每行/列作为独立 chunk | 适合大型表格 |

**推荐方案**：表格摘要 + 原始表格保留。对表格生成一段自然语言摘要（描述表格结构、关键数据点），对摘要做 Embedding 用于检索。检索命中后，将原始表格一并返回给 LLM。

**图片处理**：

| 方法 | 说明 |
|------|------|
| OCR 提取 | 对含文字的图片做 OCR，提取文本后走标准 RAG 流程 |
| 图片描述（Captioning） | 用多模态模型（如 LLaVA、GPT-4V）为图片生成文字描述 |
| 多模态 Embedding | 使用多模态 Embedding 模型（如 CLIP）对图片直接编码 |
| 混合方案 | OCR + Captioning 组合，全面覆盖图片信息 |

**推荐方案**：
1. 含文字的图片：OCR 提取文本
2. 图表/截图：用多模态模型生成描述
3. 对生成的文本做 Embedding，图片原始文件保留用于展示

**生产实践建议**：
- 文档解析阶段就识别出图片和表格，分别处理
- 为每个非文本元素生成 metadata，在检索时可以作为过滤条件
- 多模态 RAG 目前还在快速发展中，建议关注最新的 Vision-Language 模型

**加分项**：能提到具体的 OCR 工具（Tesseract、PaddleOCR），能讨论文档解析工具（Unstructured、LlamaParse），能提到多模态 Embedding 模型（如 CLIP、Jina CLIP）。

## Q19. RAG 系统的 Token 成本怎么控制？

### 考察点

对 RAG 系统生产运营中成本优化的实战经验。

### 解答思路

1. 拆解 RAG 系统的 Token 消耗构成
2. 针对每个环节给出优化策略
3. 给出量化的成本优化效果

### 参考答案

RAG 系统的 Token 成本主要来自三个环节：Query 改写、LLM 输入上下文、LLM 输出生成。

**成本拆解与优化策略**：

**1. 检索结果压缩（最大优化空间）**
- **问题**：Top-K=10 且每个 chunk=1000 tokens 时，仅上下文就是 10000 tokens
- **优化**：
  - 用 Rerank 提高精度，Top-K 从 10 降到 3-5
  - 对检索结果做 LLM 摘要（小模型），将 1000-token chunk 压缩为 200-token 摘要
  - 去除重复内容（去重去噪）

**2. Prompt 优化**
- 精简系统 Prompt，去掉冗余指令
- 使用模板缓存和 Prompt 缓存（Prompt Caching）
- 避免在 Prompt 中重复相同信息

**3. 模型分层（Model Cascading）**
```
简单问题 --> 小模型（GPT-3.5 / 7B 开源模型）
复杂问题 --> 大模型（GPT-4 / Claude）
```
通过路由判断问题复杂度，80% 的简单问题用小模型解决，成本可降低 60%-80%。

**4. 缓存策略**
- **语义缓存**：对相似问题的回答做缓存（Embedding + 相似度匹配）
- **精确缓存**：相同 Query 直接返回缓存结果
- 缓存命中率可达 20%-40%

**5. 流式输出**
- 使用 streaming 减少等待时间
- 用户可以在生成过程中提前终止（如果答案已足够）

**6. 其他优化**
- Batch API（OpenAI 的 batch 模式，成本减半，但延迟高）
- 选择性价比高的模型（如 Claude Haiku、GPT-4o-mini）
- 监控和分析 Token 使用分布，持续优化

**量化效果参考**：
| 优化措施 | 成本降低 |
|---------|---------|
| Rerank + 降低 Top-K | 30%-50% |
| 语义缓存 | 20%-40% |
| 模型分层 | 40%-60% |
| Prompt 精简 | 10%-20% |
| 组合优化 | 60%-80% |

**加分项**：能提到具体的缓存实现（如 Redis + Embedding 做语义缓存），能讨论成本与效果的平衡点分析方法。

## Q21. 如果用户的问题很模糊，检索效果差，怎么办？

### 考察点

对边界情况处理的工程思维和人机交互设计能力。

### 解答思路

1. 识别模糊问题的特征
2. 给出多层次的处理策略
3. 强调主动交互的重要性

### 参考答案

模糊问题是 RAG 系统中最常见也最难处理的情况。处理的核心原则是：**与其返回一个错误的答案，不如引导用户给出更清晰的问题。**

**模糊问题的典型特征**：
- 缺少关键实体（"这个怎么做？"）
- 过于宽泛（"介绍一下你们公司"）
- 包含歧义（"苹果"指的是水果还是公司？）
- 信息不完整（只说了问题背景，没说具体问题）

**处理策略（从轻到重）**：

**策略一：检索侧兜底**
- 降低检索阈值，扩大召回范围
- 使用 Step-back Prompting，将具体问题抽象化后检索
- 如果所有结果的相关度都很低，触发兜底策略

**策略二：LLM 侧澄清**
- 让 LLM 判断检索结果的相关度，如果都不相关，主动询问用户
- 提供几个可能的理解方向，让用户选择

```
LLM 输出：
"您提到的'这个'，我理解可能是以下几种情况之一：
1. 关于 XX 产品的使用方法
2. 关于 XX 功能的配置步骤
3. 关于 XX 问题的排查流程
请问您想了解的是哪个方面？"
```

**策略三：主动追问**
- 当检测到问题模糊时，不直接回答，而是反问用户
- 提供结构化选项（按钮、下拉菜单），降低用户输入成本

**策略四：意图识别路由**
- 先用一个分类模型判断问题类型
- 如果是"信息查询"类，走 RAG 流程
- 如果是"模糊/闲聊"类，直接由 LLM 回答或引导用户

**工程实现建议**：
```
Query --> 意图分类器 --> 清晰问题 --> RAG 流程
                     --> 模糊问题 --> 澄清/追问
```

**加分项**：能讨论澄清策略的 A/B 测试方法（对比直接回答 vs 追问的转化率），能提到主动学习（从用户的追问中学习什么才算"清晰的问题"）。

## Q22. RAG 系统的端到端延迟怎么优化？

### 考察点

对系统性能优化的全链路思维和实践经验。

### 解答思路

1. 拆解 RAG 系统各环节的延迟
2. 针对每个环节给出优化策略
3. 给出具体的延迟目标和优化效果

### 参考答案

RAG 系统的端到端延迟由多个环节叠加而成，需要逐一分析和优化。

**延迟拆解**（典型值，单位 ms）：

| 环节 | 典型延迟 | 占比 |
|------|---------|------|
| Query 改写 | 50-200 | 5%-10% |
| 向量检索 | 20-100 | 5%-15% |
| BM25 检索 | 5-20 | 1%-3% |
| 结果融合 | 1-5 | <1% |
| Rerank | 100-500 | 15%-30% |
| Prompt 组装 | 1-5 | <1% |
| LLM 生成（首 token） | 500-2000 | 40%-60% |
| LLM 生成（完整） | 1000-5000 | - |

**优化策略**：

**1. 检索层优化**
- 向量检索使用 GPU 加速（Milvus 支持 GPU 索引）
- 调整 HNSW 的 ef_search 参数，在精度和速度间取舍
- BM25 使用 Elasticsearch 的缓存
- 预计算常用 Query 的检索结果

**2. Rerank 层优化**
- 减少 Rerank 的候选数量（从 100 降到 30-50）
- 使用更轻量级的 Rerank 模型（如 BGE-Reranker-base vs large）
- 只 Rerank Top-20 而非 Top-100
- 批量 Rerank（一次性发送所有候选）

**3. LLM 层优化**
- 首 Token 延迟优化：使用 streaming 输出
- 减少上下文长度（压缩检索结果）
- 使用更快的推理引擎（vLLM、TGI）
- Prompt Caching（利用模型的上下文缓存能力）
- 模型蒸馏（用小模型达到接近大模型的效果）

**4. 系统级优化**
- 并行化：向量检索和 BM25 检索并行执行
- 语义缓存：命中缓存时跳过整个检索 + 生成链路
- 预热：热门文档预先加载到内存
- CDN：对静态回答结果做边缘缓存

**延迟优化效果参考**：

| 优化前 | 优化后 | 主要优化手段 |
|--------|--------|------------|
| 3000-5000ms | 1500-2500ms | Rerank 优化 + 并行化 |
| 1500-2500ms | 800-1500ms | 语义缓存 + Prompt 压缩 |
| 800-1500ms | 300-800ms | 小模型替代 + 全链路优化 |

**加分项**：能讨论 Speculative Decoding（推测解码）加速 LLM 生成，能提到 Speculative RAG（并行检索多个候选），能给出 Profiling 工具和优化方法论。

## Q23. RAG 系统的评测维度有哪些？如何构建评测数据集？

### 考察点

对 RAG 系统质量评估体系的系统化思维和落地能力。

### 解答思路

1. 梳理多维度评测体系
2. 说明数据集构建方法和数据量要求
3. 讨论自动化评测和人工评估的结合

### 参考答案

RAG 系统的评测需要覆盖检索层和生成层，建立多维度的质量评估体系。

**评测维度**：

**检索层评测**：

| 维度 | 指标 | 说明 |
|------|------|------|
| 召回能力 | Hit Rate@K, Recall@K | 是否找到了相关文档 |
| 排序能力 | MRR, NDCG@K | 相关文档是否排在前面 |
| 精确率 | Precision@K | 返回的文档中有多少是相关的 |
| 覆盖率 | 知识库覆盖率 | 评测集中的问题有多少能被知识库覆盖 |

**生成层评测**：

| 维度 | 指标 | 说明 |
|------|------|------|
| 忠实度 | Faithfulness | 回答是否基于检索到的上下文 |
| 相关性 | Answer Relevance | 回答是否切题 |
| 准确性 | Context Accuracy | 回答中的事实是否与上下文一致 |
| 完整性 | Completeness | 回答是否覆盖了问题的所有方面 |
| 流畅度 | Fluency | 回答的语言是否自然流畅 |
| 有用性 | Helpfulness | 回答是否真正解决了用户的问题 |

**构建评测数据集的方法**：

**步骤一：需求分析**
- 梳理用户使用场景，列出典型问题类型
- 与业务方沟通，确定核心场景和问题范围

**步骤二：问题收集**
- **历史日志**：从生产日志中提取真实用户问题（脱敏后）
- **业务专家**：让领域专家编写覆盖各种场景的问题
- **LLM 生成**：用 LLM 基于知识库自动生成问题和参考答案
- **用户调研**：收集用户反馈中的典型问题

**步骤三：标注**
- 每个问题标注：标准答案、相关文档、问题类型、难度等级
- 采用双盲标注 + 交叉校验保证质量
- 对于难以达成共识的 case，引入第三方仲裁

**步骤四：数据集划分**
- 训练/验证/测试 = 6:2:2 或 7:1.5:1.5
- 确保各类型问题在数据集中有代表性

**数据量建议**：
- 最小可用：100-200 个 QA 对
- 推荐规模：500-1000 个 QA 对
- 大型系统：2000+ 个 QA 对

**自动化评测**：
- 使用 Ragas、DeepEval 等框架自动计算指标
- LLM-as-a-Judge：用强模型（如 GPT-4）评估回答质量
- 定期（每周/每两周）自动运行评测，生成报告

**加分项**：能讨论 LLM-as-a-Judge 的偏差问题，能提到 Ragas 框架的具体指标计算方式，能讨论人工评估和自动化评估的权重分配。

## Q24. 你了解 Agentic RAG 吗？它和普通 RAG 有什么区别？

### 考察点

3. 给出典型架构和落地场景

### 参考答案

Agentic RAG 是 RAG 的下一代演进方向，其核心区别在于：**普通 RAG 是固定的流水线，Agentic RAG 引入了 Agent 的自主决策和动态编排能力。**

**普通 RAG 的局限**：
- 固定流程：Query --> 检索 --> 排序 --> 生成，流程不可变
- 无法处理复杂任务：多跳查询、需要外部工具的场景无能为力
- 无法自我纠错：检索不到正确信息也不会调整策略

**Agentic RAG 的核心特征**：
1. **自主决策**：Agent 自主决定是否需要检索、检索什么、检索几次
2. **工具使用**：可以调用搜索 API、数据库查询、代码执行等外部工具
3. **动态编排**：根据任务复杂度动态调整工作流
4. **自我反思**：评估检索结果和生成质量，不满足时重新检索

| 维度 | 普通 RAG | Agentic RAG |
|------|---------|-------------|
| 流程 | 固定流水线 | 动态编排 |
| 检索决策 | 每次都检索 | 自主判断是否需要 |
| 检索次数 | 固定一次 | 多次迭代 |
| 工具能力 | 无 | 可调用外部工具 |
| 错误处理 | 无 | 自我反思和重试 |
| 适用场景 | 简单问答 | 复杂推理、多步骤任务 |
| 实现框架 | LangChain Pipeline | LangGraph、AutoGen、CrewAI |
| 延迟 | 低 | 较高（多轮交互） |
| 成本 | 低 | 较高 |

**典型 Agentic RAG 架构**：

```
用户问题 --> Agent（决策中心）
                │
        ┌───────┼───────┐
        │       │       │
   检索工具  搜索工具  代码工具
   (向量库)  (Web)    (Python)
        │       │       │
        └───────┼───────┘
                │
         Agent 整合信息 --> 评估质量
                │
          ┌─────┴─────┐
        满意        不满意
          │           │
       输出回答    重新规划检索
```

**落地场景**：
- 技术文档问答 + 代码执行验证
- 研究报告生成（需要多次检索、交叉验证）
- 数据分析问答（需要查询数据库、运行代码）

**加分项**：能提到 LangGraph 的状态图编排能力，能讨论 Agentic RAG 的延迟和成本挑战，能给出从普通 RAG 渐进式升级到 Agentic RAG 的路线图。

## Q25. 你们项目用的什么 Embedding 模型？为什么选它？和 OpenAI 的 ada-002 对比过吗？

### 考察点

对 Embedding 模型生态的了解深度，以及模型选型的系统性方法。

### 解答思路

1. 说明模型选型的评估维度
2. 对比主流 Embedding 模型
3. 给出具体的评测方法和数据

### 参考答案

Embedding 模型是 RAG 系统的基石，选型需要从多个维度综合评估。

**主流 Embedding 模型对比**：

| 模型 | 维度 | 最大长度 | 优势 | 劣势 |
|------|------|---------|------|------|
| text-embedding-ada-002 | 1536 | 8191 | API 简单易用 | 闭源、中文弱、无法本地部署 |
| text-embedding-3-large | 3072 | 8191 | 效果最强 | 贵、闭源 |
| text-embedding-3-small | 1536 | 8191 | 性价比高 | 中文效果一般 |
| BGE-m3 | 1024 | 8192 | 多语言、开源、中文强 | 需要自行部署 |
| GTE-Qwen2-7B | 3584 | 32768 | 效果接近 3-large | 需要 GPU 部署 |
| E5-mistral-7B | 4096 | 4096 | 开源、效果好 | 需要 GPU 部署 |
| Jina-embeddings-v3 | 1024 | 8192 | 多语言、免版税 | 社区较小 |

**选型评估维度**：

1. **效果**：在业务数据集上的 Hit Rate、MRR、NDCG
2. **语言支持**：中文能力是否足够强
3. **部署**：是否支持本地部署（数据隐私要求）
4. **成本**：API 调用费用 vs 自建 GPU 成本
5. **兼容性**：是否与现有向量数据库兼容
6. **上下文长度**：能否处理长文档

**对比评测方法**：
```
1. 构建 200+ 个业务 QA 对作为 Golden Dataset
2. 用每个模型做检索评估（Hit Rate@5, MRR, NDCG@10）
3. 对比端到端回答质量（LLM-as-a-Judge 评分）
4. 对比延迟和成本
```

**典型评测结果**（中文场景）：
- BGE-m3：Hit Rate@5 = 82%, MRR = 0.73
- ada-002：Hit Rate@5 = 68%, MRR = 0.58
- E5-mistral-7B：Hit Rate@5 = 85%, MRR = 0.76

**推荐理由**：
- 如果团队有 GPU 资源且需要中文支持 --> BGE-m3 或 GTE-Qwen2
- 如果追求快速上线且对中文要求不高 --> text-embedding-3-small
- 如果有数据合规要求（不能出境） --> 必须开源模型本地部署

**加分项**：能提到 C-MTEB 中文评测榜单，能讨论 Embedding 模型微调（用业务数据做 Contrastive Fine-tuning），能提到 Matryoshka Embedding（可变维度）。

## Q26. 什么是 GraphRAG？和传统 RAG 比有什么优势？

### 考察点

对 RAG 技术前沿的理解，以及知识图谱与大模型结合的创新思维。

### 解答思路

1. 解释 GraphRAG 的核心概念
2. 对比与传统 RAG 的差异
3. 分析适用场景和局限性

### 参考答案

GraphRAG 是微软研究院在 2024 年提出的 RAG 增强方案，其核心思想是**将知识图谱（Knowledge Graph）与 RAG 相结合，利用实体之间的关系信息来提升检索和生成的质量**。

**传统 RAG 的局限性**：
- 文本被切分为独立的 chunk，丢失了 chunk 之间的关系
- 无法回答需要跨文档推理的问题
- 对实体间关系的理解完全依赖 LLM 的参数化知识

**GraphRAG 的工作原理**：

```
1. 文档解析 --> 提取实体（Entity）和关系（Relation）
2. 构建知识图谱 --> 实体为节点，关系为边
3. 社区检测（Community Detection）--> 发现文档中的主题簇
4. 为每个社区生成摘要 --> 层次化的社区摘要
5. 检索时 --> 在图谱上进行图遍历 + 社区摘要检索
```

**GraphRAG 的优势**：

| 维度 | 传统 RAG | GraphRAG |
|------|---------|----------|
| 信息组织 | 扁平的 chunk 列表 | 结构化的图（实体+关系） |
| 全局理解 | 差（只能看局部 chunk） | 好（社区摘要提供全局视图） |
| 多跳推理 | 困难 | 天然支持（图遍历） |
| 实体关系 | 隐式（依赖 LLM） | 显式（图谱中的边） |
| 摘要生成 | 不支持 | 层次化社区摘要 |
| 查询覆盖度 | 局部相关 | 全局 + 局部 |
| 构建成本 | 低 | 高（需要建图谱） |
| 延迟 | 低 | 较高 |

**GraphRAG 特别适合的场景**：
- 需要回答全局性问题（"这份文档主要讲了什么？"）
- 需要跨文档的实体关系推理（"A 和 B 之间有什么联系？"）
- 需要理解复杂的关系网络（供应链、人物关系、事件脉络）
- 需要层次化的知识探索（从宏观到微观）

**GraphRAG 的局限性**：
- 构建知识图谱成本高（实体抽取、关系抽取、图谱构建）
- 对非结构化文档的图谱构建质量依赖 NLP 能力
- 图遍历检索的延迟高于向量检索
- 不适合简单的单点问答

**生产实践建议**：对于大多数业务场景，传统 RAG（混合检索 + Rerank）已经够用。GraphRAG 适用于知识密集、关系复杂的场景（如法律文档分析、金融研报分析、学术研究辅助）。

**加分项**：能提到微软 GraphRAG 的具体实现（基于 LLM 的实体关系抽取 + Leiden 社区检测），能讨论 HybridRAG（Graph + Vector 双索引）的方案。

## 附录：RAG 核心概念速查表

| 概念 | 说明 |
|------|------|
| Naive RAG | 最基本的 RAG：分块 --> 向量化 --> 检索 --> 生成 |
| Advanced RAG | 加入了预处理（Query 改写）、后处理（Rerank）等优化 |
| Modular RAG | 模块化设计，各环节可插拔、可编排 |
| Agentic RAG | 引入 Agent 自主决策的 RAG |
| GraphRAG | 基于知识图谱的 RAG |
| Self-RAG | 引入自我反思令牌的 RAG |
| CRAG | 带检索质量纠正机制的 RAG |
| HyDE | 假设文档嵌入，先生成假设答案再检索 |
| RRF | 倒数排名融合，多路检索结果融合算法 |
| NDCG | 归一化折损累积增益，检索排序评估指标 |
| MRR | 平均倒数排名，检索排序评估指标 |
| NIAH | 大海捞针测试，评估长上下文模型的精度 |
| HNSW | 分层可导航小世界，向量索引算法 |
| IVF | 倒排文件索引，向量索引算法 |
| Bi-Encoder | 双编码器，检索用（独立编码，余弦相似度） |
| Cross-Encoder | 交叉编码器，重排用（联合编码，注意力交互） |
